# docs/state-machine.md

本文件定义系统中所有关键状态机的状态集合、允许的迁移路径、触发条件、禁止迁移情形以及与前端展示和后端事务的对应关系。它的作用是把“什么状态可以走到什么状态”一次写死，避免后续实现时出现订单乱跳、航班乱改、会员等级与积分不一致、前端按钮误开放等问题。

本项目的状态机只围绕已经确认的业务规则展开，不引入新的业务实体，也不擅自扩展未确认的状态。若实现中需要临时过程态，必须限定为服务层内部变量，不得写入数据库状态字段。

## 1. 总原则

状态机设计遵循以下原则。第一，任何状态变化都必须有明确触发源，不能由前端自行修改。第二，所有写入数据库的状态变化都必须经过后端 Service 层事务处理。第三，状态迁移必须单向、可追踪、可回滚，避免循环跳转。第四，状态字段与前端文案必须区分开，前端显示文案可以更友好，但数据库中的状态值必须固定、唯一、稳定。第五，若订单、库存、积分、会员等级之间存在联动，则必须在同一事务内完成，保证 ACID 一致性。

## 2. TicketSale 状态机

TicketSale 是整个系统最重要的业务状态机。它描述了一张票从创建、支付、过期到退票或改签完成的全过程。当前确认的核心状态为：待支付、已支付、已过期、退款成功、改签成功。若实现时保留枚举英文值，建议统一为 `PENDING_PAYMENT`、`PAID`、`EXPIRED`、`REFUND_SUCCESS`、`CHANGE_SUCCESS`。前端页面可以显示为“待支付”“已支付”“已过期”“退款成功”“改签成功”，但数据库中的状态值必须固定。

TicketSale 的允许迁移关系如下：

`PENDING_PAYMENT → PAID`

`PENDING_PAYMENT → EXPIRED`

`PAID → REFUND_SUCCESS`

`PAID → CHANGE_SUCCESS`

其中，`PENDING_PAYMENT → PAID` 表示用户在支付窗口内完成模拟支付；`PENDING_PAYMENT → EXPIRED` 表示订单超时未支付，由定时任务或支付接口再次校验后标记为过期；`PAID → REFUND_SUCCESS` 表示订单已支付后办理退票成功；`PAID → CHANGE_SUCCESS` 表示原票在改签完成后被旧票状态终结。已过期、退款成功、改签成功都视为终态，原则上不再回到前述状态。

TicketSale 的禁止迁移包括：

`EXPIRED → PAID`

`REFUND_SUCCESS → PAID`

`CHANGE_SUCCESS → PAID`

`REFUND_SUCCESS → EXPIRED`

`CHANGE_SUCCESS → EXPIRED`

也就是说，一旦订单过期、退票成功或改签成功，就不能再回到可支付状态；如果系统需要重新购票，必须创建新的 TicketSale 记录，而不是复用旧记录。

### 2.1 待支付状态 `PENDING_PAYMENT`

该状态表示订单已创建、库存已锁定、支付截止时间已写入，但用户尚未完成支付。该状态下可以执行的操作只有支付和超时释放检查。前端应显示倒计时，后端应允许支付接口在截止时间前将其转入 `PAID`。

### 2.2 已支付状态 `PAID`

该状态表示模拟支付已成功，订单正式确认，票务已出票，积分已累计。该状态下允许退票和改签发起。若需要展示出票时间，则出票时间应在进入该状态时同步写入。

### 2.3 已过期状态 `EXPIRED`

该状态表示订单未在 15 分钟支付窗口内完成支付，库存已回补，订单失效。该状态是终态，不能被再次支付，也不能直接退票或改签。前端应将该状态解释为“支付超时已失效”。

### 2.4 退款成功状态 `REFUND_SUCCESS`

该状态表示支付后的订单已经成功退票。此时库存已回补，积分已回退，会员等级已重新判断。该状态是终态，不能再支付、不能再退票、不能再改签。

### 2.5 改签成功状态 `CHANGE_SUCCESS`

该状态表示原票已完成改签流程，旧票不再代表当前有效行程。原票与新票之间必须通过 `OriginalTicketId` 建立链式追踪。该状态是终态，不能再支付、不能再退票、不能再改签。

## 3. 改签流程中的双票状态关系

改签不是简单地把一张票改掉，而是“旧票终止 + 新票生成 + 差价支付”的组合行为。为了保证历史可追溯，改签后至少会同时存在两张票：原票和改签票。

原票的状态从 `PAID` 变为 `CHANGE_SUCCESS`。新票在创建时进入 `PENDING_PAYMENT`，若存在差价，则必须在 15 分钟内完成补差价支付；若差价为 0，则可以直接确认成功。新票支付成功后，新票状态转为 `PAID`，而原票仍保持 `CHANGE_SUCCESS`。

若连续改签发生多次，则每一次改签都会生成一张新的票，并让新票的 `OriginalTicketId` 指向上一张票，而不是最初那张票。这样可形成链式结构：票 A → 票 B → 票 C。前端在展示改签历史时，应按链式顺序展示，而不是只显示最初与最新两个节点。

## 4. Flight 状态机

Flight 只需要表达航班级别的运营状态。已确认的状态包括：正常、取消、延误、已完成、停用。若以英文枚举表示，建议统一为 `NORMAL`、`CANCELLED`、`DELAYED`、`COMPLETED`、`DISABLED`。

Flight 状态的含义如下：

`NORMAL` 表示航班正常可售、可显示、可用于生成可售航段。

`DELAYED` 表示航班处于延误状态，但历史订单仍保留，前台可以展示延误标识。

`CANCELLED` 表示航班取消，后续不应继续售卖该航班的新订单。

`COMPLETED` 表示航班已完成飞行，历史记录保留，通常不再用于新增订单。

`DISABLED` 表示管理员停用该航班。停用不是物理删除，已有票务记录时也不允许物理删除，只能停用。

Flight 的允许迁移关系建议如下：

`NORMAL → DELAYED`

`NORMAL → CANCELLED`

`NORMAL → COMPLETED`

`NORMAL → DISABLED`

`DELAYED → COMPLETED`

`DELAYED → CANCELLED`

`DELAYED → DISABLED`

`CANCELLED → DISABLED`

`COMPLETED → DISABLED`

一旦进入 `COMPLETED` 或 `CANCELLED`，是否允许回到 `NORMAL` 不建议开放。课程设计场景下，为避免状态回退带来的理解混乱，建议不实现回退。

Flight 状态的禁止迁移包括：

`DISABLED → NORMAL`

`COMPLETED → NORMAL`

`CANCELLED → NORMAL`

如果业务上确实需要恢复，只能通过管理员重新新增或单独创建业务操作，但不建议在课程设计中加入这种复杂分支。

## 5. MemberLevel 状态机

会员等级仅包含普通与 VIP 两级。若使用英文枚举，建议统一为 `NORMAL` 和 `VIP`。这不是严格意义上的复杂状态机，但它和积分联动，因此必须明确迁移条件。

`NORMAL → VIP`

触发条件为积分达到或超过 1000 分。

`VIP → NORMAL`

触发条件为积分在退票或积分回退后低于 1000 分。

会员等级迁移必须由积分变化驱动，不能由管理员手工修改，不能由前端直接选择。

会员等级只与当前积分值相关，系统不保留复杂会员历史，也不引入多级会员体系。这样实现最简单，也最适合演示。

## 6. Points 与 MemberLevel 联动规则

积分不是独立状态机，但它直接决定会员等级，因此必须在状态机文档中说明。

购票成功后，用户积分增加 100 分。退票成功后，用户积分扣回 100 分。若积分变更后达到 1000 分或以上，则会员等级自动转为 VIP；若积分回退后低于 1000 分，则会员等级自动恢复为普通。

积分变化和会员等级变化必须在同一事务中提交，不能出现积分已经变更但会员等级尚未更新的中间态。

前端展示时应同步显示积分与会员等级的变化，尤其是在“900 分用户买一张票后升级 VIP，再买一张票享受九折”的演示流程中，这一状态联动必须清晰可见。

## 7. PaymentWindow 状态机视角

支付窗口不是数据库持久化状态，但它是订单状态机的重要约束条件，因此必须单独说明。

支付窗口从订单创建时开始计算，固定为 15 分钟。只要当前时间小于等于 `ExpiredAt`，订单就仍处于可支付窗口内；一旦超过 `ExpiredAt`，订单就不再可支付，且应在支付接口或定时任务中被标记为 `EXPIRED`。

支付窗口本身不会被前端写入，也不会被前端修改。前端只负责倒计时显示，后端负责最终判定。

## 8. Inventory 状态视角

库存不使用单独的状态字段，但库存会随着订单状态流转发生变化，因此也必须定义其逻辑迁移。

下单成功后，目标航段的库存从可售变为已锁定的视角表现。这里所谓“锁定”，并不是增加一个数据库状态字段，而是通过事务内行级锁和余票扣减来实现。

支付成功后，库存保持扣减结果，不再回滚。

订单过期后，库存恢复。

退票成功后，库存恢复。

改签时，新票对应的新航段库存扣减，旧票对应旧航段库存回补或旧票失效处理按照事务规则执行。

库存的本质是数值变化，不是状态值变化，但它与订单状态强相关，因此前端展示和后端事务必须保持同步。

## 9. 用户登录状态

登录态并不是数据库核心业务状态，但它决定前端是否可以访问受保护页面，因此建议在工程层面保持统一。

未登录 → 已登录

这是唯一主要迁移。

已登录 → 未登录

对应退出登录或登录态失效。

用户登录态不需要复杂分支。前端只需根据登录态决定页面跳转与菜单显示，后端接口仍需逐次鉴权。

## 10. 状态迁移中的事务要求

所有状态迁移都不是单独的字段更新，而是业务结果的最终表现。只要状态变化涉及库存、积分、会员等级、改签链或支付截止时间，就必须纳入事务。

典型要求如下：

1. 订单从 `PENDING_PAYMENT` 转 `PAID` 时，必须同时完成库存确认、积分发放、会员等级判断、支付时间记录。
2. 订单从 `PAID` 转 `REFUND_SUCCESS` 时，必须同时完成库存回补、积分回退、会员等级重判、退票时间记录。
3. 订单从 `PAID` 转 `CHANGE_SUCCESS` 时，必须同时写入原票状态、新票记录、差价支付状态和改签来源链。
4. 订单从 `PENDING_PAYMENT` 转 `EXPIRED` 时，必须同时执行库存回补与状态更新。

不能把状态更新和库存更新拆到两个独立请求里，否则会破坏一致性。

## 11. 前端展示规则

前端只负责展示状态，不负责决定状态。

对 TicketSale，前端应展示：待支付、已支付、已过期、退款成功、改签成功。

对 Flight，前端应展示：正常、延误、取消、已完成、停用。

对会员等级，前端应展示：普通、VIP。

若后台返回的状态值与前端文案不一致，前端必须以后台状态值为准，再做字典映射。不要在前端自行发明状态含义。

## 12. 实现建议

建议后端为每个状态机建立枚举类，并在 Service 层集中处理状态迁移判断。不要把状态判断散落在 Controller、Repository 或前端页面中。

建议状态更新方法统一经过一个转换函数，明确当前状态、目标状态、允许条件和拒绝原因。这样既便于单元测试，也便于后续联调排错。

如果需要在数据库中为状态字段建立约束，建议使用 `VARCHAR` 配合程序枚举映射，或者使用 `TINYINT` 配合字典表，但必须保证可读性和稳定性。课程设计场景下，优先保证实现简单与字段含义清楚。

## 13. 总结

本项目最关键的状态机是 TicketSale，其次是 Flight，再次是 MemberLevel。只要这三个状态机稳定，支付、退票、改签、会员升级、航班停用和库存回补就都能稳定运行。

实现时必须坚持：

* 状态值固定
* 状态迁移单向
* 状态变更与事务绑定
* 前端只展示不决策
* 历史数据可追溯

这五条是整个系统不出错的核心。
