## 2. 开场说明

先用 30 秒说明系统范围：

```text
我们做的是航空票务数据库系统，重点不是商业级支付平台，而是数据库课程项目。
系统严格围绕 ER 图实现，核心表包括用户、城市、机场、飞机、航班、航段、售票记录、餐食、餐食预订。
业务上覆盖管理员维护航班资源、乘客查票下单、支付、退票、改签、会员升级和库存恢复。
```

然后强调三个数据库重点：

```text
第一，FlightSegment 是最小售卖单元，库存挂在航段上，不挂在航班上。
第二，下单时使用事务和行级锁，避免并发超卖。
第三，订单状态只在后端服务层流转，库存、积分、会员等级在同一事务里更新。
```

```text

向老师说明：

### 3.2 查询航班

重点让老师看到：

- 到达机场

```

### 3.3 新增航班

在航班管理页填写一条演示航班，例如：

```text
航班号：MU9999
日期：2026-07-01
飞机：选择任意可用飞机
状态：NORMAL
出发机场：PEK
到达机场：SHA
```

点击保存。

说明：

```text
新增航班会写入 Flight 表，并通过外键关联 Aircraft 和 Airport。
```

### 3.4 修改航班

将刚才的航班状态从 `NORMAL` 改为 `DELAYED`。

说明：

```text
这里演示 Update 操作。状态值严格限定在 ER 和业务规则允许的枚举中，不额外发明状态。
```

### 3.5 停用航班

点击停用航班，使状态变为 `DISABLED`。

说明：

```text
这里不做物理删除，而是软停用。
原因是航班可能已经被订单引用，直接删除会破坏外键完整性和历史订单可追溯性。
```

### 3.6 航段维护

进入航段管理页，展示航段列表。

重点说明：

```text
航段是最小售卖单元。比如一个航班 A -> B -> C，可以售卖 A -> B、B -> C、A -> C 三个航段。
每个航段有独立的头等舱余票、经济舱余票和价格。
```

展示字段：

- 起点站序
- 终点站序
- 起飞机场
- 到达机场
- 计划起飞时间
- 计划到达时间
- 经济舱余票

- 经济舱价格

## 5. VIP 升级与折扣演示

退出后登录：

```text
passengerA / pass123
```

### 5.1 展示初始会员状态

先展示会员信息：

```text
积分：900
会员等级：NORMAL

```

说明：

系统只有 NORMAL 和 VIP 两种会员等级。
每支付一张票增加 100 积分，达到 1000 分自动升级 VIP。

### 5.2 购买第一张票

按普通购票流程搜索、下单、支付一张票。

支付后展示：

```text

会员等级：NORMAL -> VIP
```

说明：

```text
积分增加和会员升级在支付事务里一起完成，避免积分和等级不一致。
```

### 5.3 购买第二张票

再次下单一张票。

重点展示：

```text
PriceAmount：数据库存储原价
PaymentAmount：VIP 用户按 0.9 折扣结算
```

例如：

```text
原价 1000，应付 900
```

说明：

```text
数据库保留原价，结算时根据会员等级计算实付金额。
这样既能保留价格历史，也能清楚展示优惠。
```

## 6. 退票演示

使用任意已支付订单，点击退票。

展示：

```text
订单状态：PAID -> REFUND_SUCCESS
库存恢复
积分回滚 100
会员等级重新判断
```

说明：

```text
退票不是只改订单状态。
系统会在同一事务中完成状态更新、库存恢复、积分回滚和会员等级重算。
```

如果使用 `passengerA`，可以顺带展示：

```text
如果积分从 1000 以下，VIP 会回退为 NORMAL。
```

## 7. 改签演示

选择一张 `PAID` 订单，点击改签。

选择另一条航班或航段，提交改签申请。

展示：

- 新订单状态：`PENDING_PAYMENT`
- 新订单的 `OriginalTicketId` 指向旧订单
- 如果有差价，显示需要补差价

支付改签订单后展示：

```text
旧票：PAID -> CHANGE_SUCCESS
新票：PENDING_PAYMENT -> PAID
OriginalTicketId：指向上一张票
```

说明：

```text
改签支持跨航班。
系统不是直接修改旧票，而是生成一张新票，并用 OriginalTicketId 形成链式追踪。
这样历史记录可追溯，符合数据库设计要求。
```

## 9. 防超卖演示话术

如果现场时间足够，可以准备一条只剩 1 张经济舱票的航段。

展示方式：

1. 查询该航段，确认经济舱余票为 1。
2. 乘客 A 下单成功，余票变为 0。
3. 乘客 B 再下同一航段，系统提示余票不足。

向老师说明：

```text
下单前后端会对 FlightSegment 执行悲观写锁，相当于 SELECT ... FOR UPDATE。
库存扣减和订单创建在同一个事务中提交。
所以即使并发请求同时到达，也只有一个请求能扣到最后一张票。
```

可以补充代码位置：

```text
FlightSegmentRepository.findByIdForUpdate()
TicketService.createTicket()
```

## 10. 安全与隐私说明

展示注册或下单时，说明：

```text
用户密码不会明文入库，使用 BCrypt 保存哈希。
身份证号不会明文保存，系统保存 SHA-256 摘要。
```

如果老师问为什么订单里看不到身份证明文：

```text
这是故意设计的。票务系统涉及个人敏感信息，数据库只保存摘要，满足演示需求同时保护隐私。
```

## 11. 结尾总结

最后用 30 秒总结：

```text
本系统完成了航空票务数据库的核心闭环：
管理员维护城市、机场、飞机、航班、航段、餐食；
乘客完成注册、登录、查询、下单、支付、查看订单；
扩展业务包括会员升级、VIP 折扣、退票、改签和支付超时释放库存。

数据库层面严格遵守 ER 图，不新增核心业务表；
业务层面通过事务和行级锁保证库存一致性；
历史数据通过订单状态和 OriginalTicketId 保持可追溯。
```

## 12. 推荐展示顺序

时间紧张时按这个顺序：

1. 管理员登录，展示概览。
2. 航班查询、新增、修改、停用。
3. 航段列表，说明航段是最小售卖单元。
4. `passengerB` 查询航班、下单、支付、查看订单。
5. `passengerA` 展示 900 积分，买第一张票升级 VIP。
6. `passengerA` 买第二张票，展示 9 折。
7. 演示退票，展示库存和积分回滚。
8. 演示改签，展示 `OriginalTicketId`。
9. 有时间再演示超时释放库存和防超卖。

## 13. 现场风险控制

- 展示前先重置演示数据。
- 不要临时修改数据库表结构。
- 不要现场手写 SQL 改核心数据。
- 不要演示还没完整验证过的路线。
- 如果某个搜索条件查不到航班，立刻切回固定条件 `PEK -> SHA, 2026-07-01`。
- 如果时间不够，优先保留“航段售票、事务库存、VIP 折扣、退票改签”四个亮点。
