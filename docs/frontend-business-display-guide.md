# 航空票务系统前端业务展示文档

本文档用于和前端同学对齐页面、业务展示、按钮状态和演示流程。

接口地址、字段名、GET/POST 规范已经由接口文档单独约定，本文不重复接口细节。前端实现时应以既有接口规范和后端返回数据为准。

## 1. 前端整体目标

本项目是数据库课程设计《航空票务系统》。前端重点不是实现复杂商业平台，而是把后端和数据库的业务闭环清晰、稳定地展示出来。

前端需要重点展示：

```text
注册
登录
退出登录
注销账号
旅客端购票流程
管理员端管理流程
会员积分累计
VIP 自动升级
VIP 9 折
订单状态变化
库存锁定与恢复
退票
改签
支付超时释放库存
```

演示优先级：

```text
稳定演示 > 页面炫酷 > 复杂交互
```

前端不要自行发明业务状态、会员等级、角色类型。所有状态值以后端返回为准。

## 2. 用户角色和页面分流

系统只有两类用户：

```text
PASSENGER：普通旅客
ADMIN：管理员
```

登录成功后，前端根据后端返回的 `userType` 自动进入不同界面。

```text
PASSENGER -> 旅客端页面
ADMIN     -> 管理员端页面
```

不要让用户手动选择“我是管理员/我是旅客”。角色由后端登录结果决定。

顶部导航建议统一展示：

```text
系统名称
当前登录账号
当前角色
会员等级，仅旅客显示
退出登录按钮
```

旅客端可以额外显示：

```text
注销账号按钮
```

管理员端不要显示旅客购票操作。

## 3. 演示账号要求

前端登录页建议提供三个快捷登录按钮，方便演示：

| 用户 | 账号 | 密码 | 用途 |
|---|---|---|---|
| 管理员 | `admin` | `admin123` | 管理员 CRUD 演示 |
| 旅客 A | `passengerA` | `pass123` | 900 分，用于演示升级 VIP 和 9 折 |
| 旅客 B | `passengerB` | `pass123` | 0 分，用于演示普通购票流程 |

旅客 A 是加分演示重点：

```text
初始积分：900
初始等级：NORMAL
第一张票支付后：1000 分，升级 VIP
第二张票下单时：展示 VIP 9 折
```

旅客 B 是普通流程演示重点：

```text
初始积分：0
用于展示普通用户注册、登录、查询、下单、支付、查看订单
```

## 4. 未登录页面

未登录时只展示登录/注册页。

### 4.1 登录区域

需要展示：

```text
登录账号输入框
密码输入框
登录按钮
演示账号快捷按钮
切换到注册
```

登录成功后：

```text
如果是 PASSENGER，进入旅客端
如果是 ADMIN，进入管理员端
```

登录失败时：

```text
展示后端返回的错误 message
不要写死“登录失败”
```

### 4.2 注册区域

注册仅面向普通旅客。

需要展示：

```text
登录账号
密码
姓名
手机号
邮箱
身份证号
注册按钮
切换到登录
```

前端需要做基础校验：

```text
账号不能为空
密码不能为空，建议至少 6 位
姓名不能为空
手机号必须是 11 位数字
邮箱格式正确
身份证号格式正确
```

注意：

```text
身份证号由用户明文输入。
前端不要加密。
后端会做 SHA-256 摘要后入库。
```

注册成功后建议：

```text
提示“注册成功，请登录”
自动切换到登录页
或自动填入刚注册账号
```

## 5. 旅客端页面结构

旅客端建议分为 5 个区域：

```text
会员信息区
航班搜索区
航班结果区
订单区
账号区
```

不建议拆得太散，否则演示时来回跳转容易出错。

## 6. 旅客端：会员信息区

会员信息区必须明显展示：

```text
用户姓名
登录账号
会员等级：NORMAL / VIP
当前积分
VIP 门槛：1000
VIP 折扣：0.9
积分进度条
```

旅客 A 登录时，页面应清楚展示：

```text
当前积分：900
距离 VIP：还差 100 分
当前等级：NORMAL
```

支付第一张票后，刷新会员信息，应展示：

```text
当前积分：1000
当前等级：VIP
```

建议做一个明显提示：

```text
恭喜升级 VIP
```

但不要改变后端状态名，状态仍然显示 `VIP`。

## 7. 旅客端：航班搜索区

搜索区需要让用户选择：

```text
出发机场
到达机场
出发日期
舱位，可选
```

建议：

```text
机场用下拉框，不要让用户手动输入机场代码
日期默认选在种子数据范围内
```

种子数据日期范围：

```text
2026-06-28 ~ 2026-07-04
```

演示推荐路线：

```text
北京 -> 上海
2026-07-01
```

搜索前校验：

```text
出发机场不能为空
到达机场不能为空
出发和到达不能相同
日期不能为空
```

搜索失败或无结果时：

```text
显示“暂无可售航班”
不要空白页面
```

## 8. 旅客端：航班结果区

每个航班结果卡片或表格行建议展示：

```text
航班号
航班日期
出发机场
到达机场
起飞时间
到达时间
经济舱余票
头等舱余票
经济舱价格
头等舱价格
航班状态
是否可售
```

可售规则展示建议：

| 航班状态 | 前端表现 |
|---|---|
| `NORMAL` | 可以下单 |
| `DELAYED` | 可显示延误标记，是否可售以后端返回为准 |
| `CANCELLED` | 不允许下单 |
| `COMPLETED` | 不允许下单 |
| `DISABLED` | 不允许下单 |

下单按钮只在可售时启用。

如果余票为 0：

```text
对应舱位不能选择
显示“无余票”
```

## 9. 旅客端：下单区域

用户选择某个航班后，显示下单表单。

需要展示：

```text
已选航班
已选航段
出发到达
日期时间
舱位选择
乘机人姓名
乘机人身份证号
餐食选择
价格预览
创建订单按钮
```

餐食至少包含：

```text
Normal Meal
Halal Meal
Vegetarian Meal
```

价格展示要注意：

普通用户：

```text
原价：980
实付：980
```

VIP 用户：

```text
原价：980
VIP 折扣：9 折
实付：882
```

重点：

```text
前端可以展示价格，但最终价格以后端创建订单返回结果为准。
不要让前端自己决定支付金额。
```

创建订单成功后，应立即展示：

```text
订单状态：PENDING_PAYMENT
订单号
应付金额
支付倒计时
支付按钮
```

## 10. 旅客端：订单区

订单区是演示重点，必须清楚展示状态变化。

订单列表建议展示：

```text
订单号
航班号
出发到达
航班日期
起飞时间
舱位
乘机人
餐食
原价
实付金额
订单状态
下单时间
支付时间
过期时间
```

订单状态只允许显示：

```text
PENDING_PAYMENT
PAID
EXPIRED
REFUND_SUCCESS
CHANGE_SUCCESS
```

不要自行改成其他状态，例如：

```text
已取消
已关闭
已退订
```

可以用中文辅助说明，但原始状态值最好也显示出来，方便老师看到数据库状态流转。

例如：

```text
PAID（已支付）
REFUND_SUCCESS（退票成功）
```

## 11. 订单按钮显示规则

不同状态显示不同按钮。

| 订单状态 | 前端按钮 |
|---|---|
| `PENDING_PAYMENT` | 显示“支付”按钮，显示剩余支付时间 |
| `PAID` | 显示“退票”按钮，显示“改签”按钮 |
| `EXPIRED` | 不显示操作按钮，只显示“已过期” |
| `REFUND_SUCCESS` | 不显示操作按钮，只显示“已退票” |
| `CHANGE_SUCCESS` | 不显示支付/退票按钮，显示“已改签” |

支付成功后：

```text
刷新订单列表
刷新会员信息
刷新航班库存
```

退票成功后：

```text
刷新订单列表
刷新会员信息
刷新航班库存
```

改签成功后：

```text
旧票显示 CHANGE_SUCCESS
新票显示 PAID
显示 OriginalTicketId 关联关系
```

## 12. VIP 升级和打折展示

这是前端必须重点做清楚的加分项。

### 12.1 第一次购票

使用旅客 A：

```text
passengerA / pass123
```

登录后显示：

```text
积分：900
等级：NORMAL
```

购买并支付第一张票后，页面要展示：

```text
积分：900 -> 1000
等级：NORMAL -> VIP
```

建议在会员卡片上明显显示：

```text
已升级 VIP
```

### 12.2 第二次购票

再次创建订单时，价格区域要展示：

```text
原价：1000
VIP 折扣：0.9
实付：900
```

这部分必须让老师一眼看到：

```text
VIP 折扣确实生效
```

不要只显示一个最终金额，否则看不出折扣逻辑。

## 13. 退票展示

退票入口只对 `PAID` 订单显示。

点击退票前建议二次确认：

```text
确认退票？
退票后订单状态将变为 REFUND_SUCCESS，库存将恢复。
```

退票成功后展示：

```text
订单状态：REFUND_SUCCESS
退款时间
库存已回补，可重新查询航班验证
积分已回退
```

注意：

```text
本项目规则是已支付订单可随时退票。
前端不要显示“起飞前 24 小时不可退”等额外规则。
```

## 14. 改签展示

改签入口只对 `PAID` 订单显示。

改签流程建议：

```text
1. 在订单上点击“改签”
2. 展示可改签航班搜索区
3. 用户选择新航班/新航段
4. 生成改签待支付订单
5. 显示差价
6. 支付改签订单
7. 展示旧票和新票状态
```

展示重点：

```text
旧票：CHANGE_SUCCESS
新票：PAID
新票 originalTicketId 指向旧票
```

差价展示：

```text
原票价格
新票价格
需补差价
```

如果新票更便宜：

```text
差价显示为 0
```

不要显示负数差价。

## 15. 支付超时释放库存展示

这是加分项，可以放在管理员端或演示专用区域。

前端展示逻辑：

```text
1. 找一个库存较少的航段
2. 创建订单但不支付
3. 显示订单状态 PENDING_PAYMENT
4. 显示库存已减少
5. 等待过期，或管理员手动触发过期订单扫描
6. 订单变为 EXPIRED
7. 库存恢复
```

如果前端做倒计时，注意：

```text
倒计时只是显示。
真正是否过期以后端状态为准。
```

倒计时结束后应重新请求订单列表。

## 16. 账号区：退出登录和注销账号

### 16.1 退出登录

退出登录只是清除当前登录态。

前端操作：

```text
清除 localStorage 中当前用户
回到登录页
```

退出登录不影响数据库用户。

### 16.2 注销账号

注销账号是业务操作，应只对旅客显示。

展示建议：

```text
注销账号按钮
二次确认弹窗
提示注销后无法继续登录，但历史订单不物理删除
```

确认文案建议：

```text
确认注销当前旅客账号？
注销后该账号将不可继续使用，历史订单仍保留用于系统记录。
```

注销成功后：

```text
清空登录态
回到登录页
显示“账号已注销”
```

管理员账号不显示注销入口。

## 17. 管理员端页面结构

管理员端建议分 5 个 Tab：

```text
概览
航班管理
航段管理
资源管理
用户与订单
```

管理员端主要用于展示：

```text
新增航班
查询航班
修改航班
停用航班
恢复航班
查看订单
查看用户
手动触发过期订单扫描
```

管理员端不要提供旅客购票按钮。

## 18. 管理员端：概览页

概览页建议展示统计卡片：

```text
用户数量
航班数量
航段数量
订单数量
已支付订单数量
退票数量
改签数量
VIP 用户数量
```

需要有一个按钮：

```text
手动触发过期订单扫描
```

用于演示：

```text
PENDING_PAYMENT -> EXPIRED
库存恢复
```

操作后刷新：

```text
订单列表
航段库存
统计卡片
```

## 19. 管理员端：航班管理

航班管理必须支持：

```text
新增航班
查询航班
修改航班
停用航班
恢复航班
```

航班列表展示：

```text
航班号
日期
飞机编号
出发机场
到达机场
航班状态
是否可售
创建时间
操作按钮
```

状态只允许：

```text
NORMAL
DELAYED
CANCELLED
COMPLETED
DISABLED
```

停用航班时，按钮文案建议用：

```text
停用
```

不要写：

```text
删除
```

因为系统要求保留历史数据，不做物理删除。

停用前二次确认：

```text
确认停用该航班？
停用后旅客端将不可继续购买该航班。
```

## 20. 管理员端：航段管理

航段是最小可售单位。前端要让管理员理解：

```text
一个航班可以有多个航段。
用户买的是某一个航段，不是整条航线。
```

航段列表展示：

```text
航班号
起点机场
终点机场
起点站序
终点站序
计划起飞时间
计划到达时间
经济舱余票
头等舱余票
经济舱价格
头等舱价格
```

编辑航段时要提醒：

```text
起点站序必须小于终点站序
经济舱余票不能小于 0
头等舱余票不能小于 0
价格不能为负
```

## 21. 管理员端：资源管理

资源管理建议包含：

```text
城市
机场
飞机
餐食
```

展示重点：

```text
城市和机场用于航班搜索
飞机用于创建航班
餐食用于旅客下单选择
```

餐食需要展示是否可用：

```text
Normal Meal
Halal Meal
Vegetarian Meal
```

如果餐食停用：

```text
旅客端不应再作为可选餐食展示
```

城市、机场、航段不建议做物理删除按钮。如后端没有停用字段，前端只做查看或编辑，不做删除。

```text
用户账号
姓名
角色
会员等级
积分
手机号
邮箱
账号状态
```

注意：

```text
不要展示 passwordHash
不要展示 idNumberDigest
不要展示身份证明文
```

订单列表展示：

```text
订单号
用户
航班
航段
舱位
订单状态
原价

下单时间
```

管理员可以查看所有订单，但不建议随意修改订单状态。订单状态应主要由旅客操作和后端业务流程改变。

## 23. 全局交互要求

所有操作都要有明确反馈：

```text
成功：显示成功提示，并刷新相关数据
失败：显示后端返回的 message
加载中：按钮禁用，避免重复点击
空数据：显示空状态文案
```

按钮重复点击防护：

```text
登录中，禁用登录按钮
支付中，禁用支付按钮
退票中，禁用退票按钮
保存中，禁用保存按钮
```

常见错误提示不要写死，优先展示后端返回信息。

例如：

```text
当前航段余票不足
订单状态不允许当前操作
订单已超时，请重新下单
账号已存在
用户名或密码错误
```

## 24. 页面刷新规则

前端要特别注意：操作成功后必须重新拉取数据。

登录成功后刷新：

```text
会员信息
订单列表
基础数据
```

创建订单成功后刷新：

```text
订单列表
航班搜索结果，库存变化
```

支付成功后刷新：

```text
订单列表
会员信息，积分和等级可能变化
航班搜索结果
```

退票成功后刷新：

```text
订单列表
会员信息
航班搜索结果，库存恢复
```

管理员修改航班后刷新：

```text
航班列表
旅客端搜索结果
```

手动触发过期订单扫描后刷新：

```text
订单列表
航段库存
统计概览
```

## 25. 前端不要做的事

前端不要做这些：

```text
不要自己编造订单状态
不要自己编造航班状态
不要让用户输入机场代码，尽量用下拉框
不要自己决定支付金额
不要在前端加密身份证号
不要物理删除航班、航段、城市、机场
不要把管理员页面展示给普通旅客
不要把旅客购票按钮展示给管理员
不要展示 passwordHash
不要展示 idNumberDigest
不要只看 HTTP 状态码，要看业务成功与失败提示
```

## 26. 演示验收路径

前端最终至少要能顺利演示以下流程。

### 26.1 普通旅客流程

```text
1. 注册一个新旅客
2. 登录
3. 查看会员信息
4. 查询航班
5. 选择航班和舱位
6. 创建订单
7. 查看 PENDING_PAYMENT
8. 支付
9. 查看 PAID
10. 查看我的订单
```

### 26.2 VIP 加分流程

```text
1. 使用 passengerA 登录
2. 展示 900 分，NORMAL
3. 买第一张票并支付
4. 展示 1000 分，VIP
5. 再买第二张票
6. 展示原价和 9 折实付金额
```

### 26.3 退票流程

```text
1. 选择 PAID 订单
2. 点击退票
3. 二次确认
4. 状态变为 REFUND_SUCCESS
5. 展示库存恢复
6. 展示积分回退
```

### 26.4 改签流程

```text
1. 选择 PAID 订单
2. 点击改签
3. 选择新航班
4. 展示差价
5. 支付改签订单
6. 旧票显示 CHANGE_SUCCESS
7. 新票显示 PAID
8. 展示 originalTicketId 关联
```

### 26.5 管理员流程

```text
1. admin 登录
2. 查看概览
3. 新增航班
4. 查询航班
5. 修改航班状态
6. 停用航班
7. 恢复航班
8. 查看订单列表
9. 手动触发过期订单扫描
```

## 27. 前端完成标准

前端完成后，至少满足：

```text
登录注册流程顺畅
旅客端和管理员端严格分离
旅客可以查航班、下单、支付、看订单
旅客 A 可以展示 900 -> 1000 -> VIP
VIP 第二单可以展示 9 折
PAID 订单可以退票
PAID 订单可以改签
管理员可以新增、查询、修改、停用航班
订单状态展示准确
金额展示准确
库存变化能通过刷新体现
失败提示能展示后端 message
演示账号快捷登录可用
```

一句话总结：

```text
前端重点不是重新设计业务，而是把后端已有业务状态清晰展示出来；
所有角色、状态、金额、积分、库存变化都以后端返回为准。
```
