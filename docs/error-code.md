docs/error-code.md
本文档定义后端统一错误码规范。所有 Controller 与 Service 返回失败时，应优先使用本文件定义的错误码，而不是随意拼接字符串消息。错误码的作用是让前端能够根据稳定代码做统一提示，也让后端开发、测试、联调、演示时便于快速定位问题。
1. 错误码总体约定
错误码格式建议为五位整数。
0：成功。
4xxxx：客户端输入错误、权限错误、资源冲突、业务状态不允许。
5xxxx：系统内部错误、数据库异常、未预期异常。
code 与 message 的关系如下：code 负责稳定识别错误类型，message 负责向前端和操作者展示自然语言提示。不要依赖 message 做程序判断。

50002：事务提交失败。
50003：事务回滚失败。
50004：未知异常。

40001：必填参数缺失。

40004：参数值非法。
40005：枚举值非法。

10. 推荐的 message 文案
错误码只需要稳定，提示文案可以简洁明确。
例如：
40102 → 用户名或密码错误。
40901 → 账号已存在。
41001 → 航班已停用，无法继续操作。
42001 → 当前航段余票不足。
42005 → 订单已超时，请重新下单。
43001 → 当前积分不足，无法完成该操作。
43003 → 会员升级失败，请稍后重试。
46001 → 过期订单扫描任务执行失败。
11. 使用规则
后端应优先在 Service 层抛出带错误码的业务异常，Controller 统一捕获后转换为 ApiResponse。
同一种错误在全项目范围内必须使用同一个 code，不得出现同义不同码。
如果一个错误既可能由参数错误导致，也可能由业务冲突导致，应优先返回更具体的错误码，不要笼统返回 40000。
对于演示场景中最常见的错误，应保证提示足够直观，例如余票不足、订单过期、航班停用、账号重复、权限不足等。
