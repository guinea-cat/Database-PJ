# 航空票务系统三人小组沟通与对接教程

本文档写给你本人使用。

你的角色是：后端开发 + 项目统筹。你不需要假装自己是很资深的项目经理，但你需要做到一件事：让前端、后端、数据库三条线始终按同一套标准工作，最后能稳定演示。

本项目最高原则：

```text
ER 图绝对优先，后端实际代码作为对接标准，演示稳定优先于复杂功能。
```

你需要反复提醒大家：

- 前端不要自己发明接口字段。
- 数据库不要自己修改 ER 图结构。
- 所有人遇到问题先定位是哪一层，不要互相猜。
- 最终目标不是做商业航空平台，而是课程数据库 PJ 稳定跑通、讲清楚、演示不翻车。

---

## 1. 你作为统筹到底要做什么

你不是要把所有人的活都干完，而是要管住 4 件事。

### 1.1 管接口

接口就是前端和后端之间的合同。

你要保证：

- 前端调用的接口路径和后端 Controller 一致。
- 前端传的字段名和 DTO 一致。
- 前端知道哪些接口用 `GET`，哪些接口用 `POST`。
- 前端知道所有响应都要判断 `code`。
- 如果前端觉得接口不够用，必须先问你，不能自己要求数据库新增字段。

你可以把这句话作为固定原则：

> 所有页面都可以自由设计，但接口字段、请求方式、状态枚举不能自由发挥。

---

### 1.2 管数据库边界

数据库边界就是 ER 图和 9 张表。

你要保证：

- 数据库同学只按 `database/schema.sql` 建表。
- 不新增核心业务表。
- 不修改主键、外键。
- 不把库存放到错误的地方。
- 不物理删除会被外键引用的数据。

你可以把这句话作为固定原则：

> 数据库可以帮我们保证数据一致性，但不能为了方便演示去改 ER 图。

---

### 1.3 管联调节奏

联调不是最后一天才开始。

你要推动大家分阶段确认：

1. 环境能跑。
2. 基础接口能通。
3. 主流程能跑。
4. 加分流程能跑。
5. 页面和演示细节打磨。

每一阶段都要有明确验收，不要只听“差不多了”。

---

### 1.4 管问题定位

三个人都缺经验时，最容易出现这种对话：

```text
前端：接口坏了。
后端：我这里没问题。
数据库：表也没问题。
```

你要把问题拆开：

```text
页面有没有发出请求？
请求路径对不对？
请求方法是 GET 还是 POST？
请求参数对不对？
后端有没有收到？
后端返回 code 是多少？
数据库有没有对应数据？
```

只要按顺序查，很多问题很快就能定位。

## 3. 给前端同学的沟通教程

### 3.1 你第一次发给前端同学的话术

可以直接复制：

```text
我把前后端对接文档写好了，你先看 docs/frontend-integration-guide.md。

你这边的重点不是改接口，而是按后端已有接口把页面做好、流程跑通。

最重要的规则：
1. 查询全部用 GET。
2. 登录、注册、下单、支付、退票、改签、管理员保存/停用/恢复全部用 POST。
3. 后端基础地址是 http://localhost:8080/api。
4. 所有响应都先判断 code，code=0 才算成功，失败就展示 message。
5. 登录后根据 userType 分页面：PASSENGER 进旅客端，ADMIN 进管理员端。
6. 状态枚举不要自己发明，只能用文档里的固定值。
7. 页面可以做得好看，但接口字段不能自己改。

你先按最小流程做：
登录/注册 -> 查航班 -> 下单 -> 支付 -> 我的订单。
这条通了以后，再做退票、改签、管理员端和美化。
```

---

### 3.2 前端同学最容易踩的坑

你要重点提醒这些。

| 坑 | 表现 | 你应该怎么提醒 |
|---|---|---|
| GET 写成 POST | 查询接口报错或查不到 | 查询类接口都用 URL 参数 |
| POST 忘记 JSON header | 后端拿不到参数 | 必须加 `Content-Type: application/json` |
| 只看 HTTP 状态码 | 明明失败了却当成功 | 必须判断返回 JSON 的 `code` |
| 手写机场代码 | 用户体验差，容易输错 | 用机场下拉框 |
| 下单只传 `flightId` | 后端报航段不存在或参数错误 | 下单必须同时传 `flightId` 和 `segmentId` |
| 改签支付调用普通支付 | 状态不对 | 普通订单用 `/ticket/pay`，改签单用 `/ticket/change/pay` |
| 订单列表只显示订单号 | 用户看不懂 | 必须显示日期、航线、时间、金额、状态 |
| 管理员端做物理删除 | 破坏外键和 ER 图 | 删除统一做停用或提示不可删除 |
| 展示敏感字段 | 页面出现密码哈希、身份证摘要 | 管理员用户列表不要展示 `passwordHash` 和 `idNumberDigest` |

---

### 3.3 你应该给前端同学的开发顺序

不要让他一开始就做完整大页面。你让他按下面顺序。

#### 第 1 步：先做统一 API 封装

目标：

- 有 `apiGet`。
- 有 `apiPost`。
- 统一处理 `code !== 0`。
- 基础地址固定 `http://localhost:8080/api`。

验收方式：

- 能调用 `GET /api/admin/airport/list`。
- 能调用 `POST /api/auth/login`。
- 失败时页面能显示后端 `message`。

你可以问他：

```text
你现在所有接口是不是都走同一个 apiGet/apiPost？有没有哪个页面自己单独写 fetch？
```

如果他说有页面自己写了 `fetch`，你让他收回来统一封装。

---

#### 第 2 步：做登录注册页

必须实现：

- 账号登录。
- 乘客注册。
- 注册前端校验。
- 登录后按 `userType` 分流。

验收方式：

- `admin/admin123` 登录后进入管理员端。
- `passengerA/pass123` 登录后进入旅客端。
- 新乘客注册成功后进入旅客端。
- 错误密码能显示错误提示。

你可以问他：

```text
你登录后是根据 userType 判断页面的吗？还是写死进某个页面？
```

如果写死，必须改。

---

#### 第 3 步：做旅客端主流程

必须实现：

1. 加载机场列表。
2. 出发机场、到达机场用下拉框。
3. 查询航班。
4. 选择航班卡片。
5. 选择舱位。
6. 填乘机人信息。
7. 创建订单。
8. 支付订单。
9. 刷新会员积分和订单列表。

验收方式：

```text
passengerA 登录
PEK -> SHA
日期 2026-07-01
查到航班
下单
支付
积分从 900 变 1000
会员等级变 VIP
```

你可以问他：

```text
你支付成功后有没有重新请求 /member/profile 和 /ticket/my？
```

因为支付后积分、会员等级、订单状态都会变，前端必须刷新。

---

#### 第 4 步：做订单展示

订单列表必须显示：

- 订单号
- 航班号
- 日期
- 出发机场
- 到达机场
- 起飞时间
- 到达时间
- 舱位
- 状态
- 原价
- 实付

你要明确告诉他：

> 订单列表不要只显示一串订单号。老师看演示时必须能一眼看懂这张票从哪到哪、什么时候飞、多少钱、什么状态。

---

#### 第 5 步：做退票和改签

退票规则：

- 只对 `PAID` 订单显示退票按钮。
- 调 `POST /api/ticket/refund`。
- 成功后刷新订单和会员。

改签规则：

- 只对 `PAID` 订单显示改签入口。
- 先选新航班。
- 调 `POST /api/ticket/change/apply` 创建改签单。
- 如果新票是 `PENDING_PAYMENT`，再调 `POST /api/ticket/change/pay` 支付差价。
- 改签完成后旧票状态是 `CHANGE_SUCCESS`，新票状态是 `PAID`。

你可以问他：

```text
你改签后有没有把旧票和新票都展示出来？有没有显示 originalTicketId？
```

---

#### 第 6 步：做管理员端

管理员端不用做得特别复杂，但要能演示：

- Dashboard 数量。
- 航班查询。
- 航班新增/修改。
- 航班停用。
- 航班恢复。
- 航段新增/修改。
- 城市、机场、飞机、餐食简单维护。
- 用户列表。
- 订单列表。

重点提醒：

```text
管理员端不要做 DELETE。
City、Airport、FlightSegment 没有停用字段，只提示不执行破坏性删除。
Flight 可以 DISABLED，也可以恢复 NORMAL。
MealOption 可以 IsAvailable=false。
Aircraft 可以 Status=DISABLED。
```

### 3.4 前端美观度指导

你的前端同学缺经验，你可以给他这些具体要求。

#### 页面整体风格

建议主题：

```text
航空运行台 + 机场航班信息屏 + 航线卡片
```

视觉关键词：

- 干净
- 稳定
- 有航空感
- 信息清晰
- 不要空白太多
- 不要像纯表格作业

#### 可以使用的元素

| 元素 | 用途 |
|---|---|
| 机场三字码 | 例如 `PEK`、`SHA`、`PVG`，增强航空感 |
| 航线箭头 | 展示 `北京（首都）PEK → 上海（虹桥）SHA` |
| 状态徽标 | 显示 `待支付`、`已支付`、`已退票`、`已改签` |
| 会员卡片 | 显示积分、VIP 进度、折扣 |
| 航班卡片 | 显示航班号、日期、起降时间、价格、余票 |
| 管理台侧边栏 | 管理员端切换概览、航班、航段、资源、订单 |
| 空状态提示 | 查询无结果时提示换日期或机场 |

#### 不建议的设计

- 不要大段口号文字。
- 不要只堆表格。
- 不要整页只有输入框和按钮。
- 不要每个东西都做成大卡片导致页面很散。
- 不要用太多颜色，状态色保持统一。
- 不要让按钮文字挤在一起。

#### 给他的具体 UI 验收标准

你可以这样要求：

```text
你做完后我们按这几个点看：
1. 登录页第一眼能看出是航空票务系统。
2. 旅客端没有大面积空白。
3. 航班卡片能一眼看懂从哪到哪、几点飞、多少钱、剩几张票。
4. 我的订单能一眼看懂每张票是什么行程。
5. 管理员端能快速找到航班、航段、资源、订单。
6. 手机或窄屏下文字不要挤爆。
```

## 4. 给数据库同学的沟通教程

### 4.1 你第一次发给数据库同学的话术

可以直接复制：

```text
我把后端和数据库对接文档写好了，你先看 docs/database-integration-guide.md。

数据库这边最重要的是严格按 ER 图和 schema.sql，不要为了方便新增表或改字段。

我们的后端不会自动建表，配置是 ddl-auto=none，所以必须先运行：
mysql -u root -p051130 < database\schema.sql
mysql -u root -p051130 airticket < database\seed_data.sql

你这边先确认：
1. 数据库名是 airticket。
2. 只有 9 张核心表。
3. 表字段、主键、外键、唯一约束、Check 约束和 schema.sql 一致。
4. passengerA 初始是 900 分 NORMAL。
5. 航班数据覆盖 2026-06-28 到 2026-07-04。

如果你觉得某个功能需要加表或加字段，先不要直接改，先跟我确认，因为我们必须以 ER 图为准。
```

---

### 4.2 数据库同学最容易踩的坑

| 坑 | 后果 | 你应该怎么提醒 |
|---|---|---|
| 自己新增表 | 偏离 ER 图，答辩难解释 | 不新增核心业务表 |
| 改字段名大小写 | JPA 映射失败 | 字段名必须和 `schema.sql` 一致 |
| 删除外键 | 数据一致性变差 | 外键必须保留 |
| 删除 Check 约束 | 错误状态可能入库 | 状态和库存约束必须保留 |
| 把库存放到 `Flight` | 业务逻辑错误 | 库存属于 `FlightSegment` |
| 物理删除用户/航班/订单 | 破坏历史记录和外键 | 用状态或匿名化 |
| 明文存密码 | 不符合安全要求 | 密码必须 BCrypt |
| 明文存身份证 | 不符合安全要求 | 身份证必须摘要 |
| 改 `passengerA` 初始积分 | VIP 演示失败 | 必须保留 900 分 |
| 航班日期太少 | 前端查不到 | 保留 2026-06-28 到 2026-07-04 |

---

### 4.3 你应该给数据库同学的执行顺序

#### 第 1 步：初始化数据库

让他执行：

```powershell
mysql -u root -p051130 < database\schema.sql
mysql -u root -p051130 airticket < database\seed_data.sql
```

验收：

```powershell
mysql -u root -p051130 -e "USE airticket; SHOW TABLES;"
```

必须只看到 9 张核心表。

---

#### 第 2 步：检查表结构

让他重点检查：

```powershell
mysql -u root -p051130 -e "USE airticket; DESC City; DESC Airport; DESC Aircraft; DESC Flight; DESC FlightSegment; DESC TicketSale;"
```

不要要求他一次看懂所有约束，先确认字段没有缺。

---

#### 第 3 步：检查外键和约束

重点确认：

- `Airport.CityId` 引用 `City.CityId`。
- `Flight.AircraftRegNo` 引用 `Aircraft.AircraftRegNo`。
- `Flight.DepartureAirportCode` 和 `Flight.ArrivalAirportCode` 引用 `Airport.AirportCode`。
- `FlightSegment.FlightId` 引用 `Flight.FlightId`。
- `TicketSale.UserId` 引用 `User.UserId`。
- `TicketSale.SegmentId` 引用 `FlightSegment.SegmentId`。
- `TicketSale.OriginalTicketId` 自关联 `TicketSale.TicketId`。
- `MealReservation.TicketId` 引用 `TicketSale.TicketId`。
- `MealReservation.MealId` 引用 `MealOption.MealId`。

你可以告诉他：

> 外键不是麻烦，它是我们数据库课程设计的得分点。

---

#### 第 4 步：检查种子数据

让他执行：

```powershell
mysql -u root -p051130 -e 'USE airticket; SELECT LoginAccount, UserType, Points, MemberLevel FROM `User`; SELECT COUNT(*) AS FlightCount FROM Flight; SELECT COUNT(*) AS SegmentCount FROM FlightSegment;'
```

必须确认：

- 有 `admin`。
- 有 `passengerA`。
- 有 `passengerB`。
- `passengerA` 是 `900 / NORMAL`。
- 航班和航段数量不是 0。

---

#### 第 5 步：配合后端联调

让他先不要改表，只观察后端能不能连。

你启动后端后，如果能访问：

```text
http://localhost:8080/api/admin/airport/list
```

就说明后端已经能查数据库。

---

### 4.4 数据库变更必须走确认流程

如果数据库同学说“我想改一下表”，你不要立刻同意。

你按下面 5 个问题问：

1. 这个改动 ER 图里有没有？
2. 不改表能不能通过已有字段实现？
3. 这个字段是展示用、计算用，还是必须落库？
4. 改了以后后端 Entity、DTO、接口会不会受影响？
5. 改了以后种子数据和演示流程会不会受影响？

只有当你确认“ER 图允许，并且后端也同步改”时，才可以动。

大多数情况下，你应该回答：

```text
这个需求先不要改表，我们从现有字段派生或在前端展示层处理。
```

## 5. 三人分阶段跟进计划

下面这个计划你可以直接用来安排进度。

---

### 阶段 0：分工确认

时间：马上，10 到 20 分钟。

你要讲清楚：

| 人 | 负责内容 | 不负责内容 |
|---|---|---|
| 你 | 后端、接口、业务规则、联调统筹 | 不负责前端所有样式细节 |
| 前端同学 | 页面、交互、接口调用、展示效果 | 不改接口、不改数据库 |
| 数据库同学 | MySQL 建表、约束、种子数据、数据检查 | 不改 ER 图、不改接口 |

会议结束时，每个人必须知道自己下一步干什么。

---

### 阶段 1：环境跑通

建议时间：半天。

目标：

- 数据库能初始化。
- 后端能启动。
- 前端能打开。

验收清单：

| 检查项 | 负责人 | 验收方式 |
|---|---|---|
| MySQL 可连接 | 数据库 | `mysql -u root -p051130 -e "SELECT VERSION();"` |
| 9 张表存在 | 数据库 | `SHOW TABLES;` |
| 种子数据存在 | 数据库 | 查 `User`、`Flight` |
| 后端启动 | 你 | `mvn spring-boot:run` |
| 后端能查机场 | 你 | 浏览器访问 `/api/admin/airport/list` |
| 前端能启动 | 前端 | 打开 `http://localhost:5173` |
| 前端能调接口 | 前端 | 页面能显示机场列表或登录成功 |

阶段通过标准：

```text
三台本地环境至少有一台完整跑通数据库 + 后端 + 前端。
```

---

### 阶段 2：最小主流程跑通

建议时间：1 天。

目标：

```text
登录 -> 查航班 -> 下单 -> 支付 -> 我的订单
```

验收步骤：

1. `passengerA/pass123` 登录。
2. 查询 `PEK -> SHA`，日期 `2026-07-01`。
3. 选择一个航段。
4. 创建订单。
5. 支付订单。
6. 查看我的订单，订单状态是 `PAID`。
7. 查看会员，积分从 900 到 1000，等级到 VIP。

阶段通过标准：

```text
不看代码，只通过页面操作能完整跑通。
```

---

### 阶段 3：扩展流程跑通

建议时间：1 天。

目标：

- 注册。
- VIP 第二张票 9 折。
- 退票。
- 改签。
- 管理员停用/恢复航班。
- 管理员保存航段。

验收步骤：

| 流程 | 通过标准 |
|---|---|
| 注册 | 新账号能注册并进入旅客端 |
| VIP 折扣 | VIP 用户第二张票 `paymentAmount = priceAmount * 0.9` |
| 退票 | 已支付订单变 `REFUND_SUCCESS` |
| 改签 | 旧票 `CHANGE_SUCCESS`，新票 `PAID` |
| 航班停用 | 停用后旅客端不可售 |
| 航班恢复 | 恢复后状态回到 `NORMAL` |

## 8. 联调排错方法

遇到问题时按下面顺序查，不要跳。

### 8.1 第一步：确认是哪一层的问题

```text
前端页面问题
接口请求问题
后端业务问题
数据库数据问题
```

判断方法：

| 现象 | 可能是哪层 |
|---|---|
| 页面按钮点了没反应 | 前端页面 |
| 浏览器 Network 没请求 | 前端事件没绑 |
| 请求地址错 | 前端 API 封装 |
| 请求是 POST 但接口要求 GET | 前端请求方法 |
| 返回 `code != 0` | 后端业务校验或数据库数据 |
| 返回 `50000` | 后端异常，需要看后端日志 |
| 后端启动失败 | 数据库连接或后端配置 |
| 查不到航班 | 数据库种子数据、查询参数或航班状态 |

---

### 8.2 第二步：看浏览器 Network

让前端同学打开浏览器开发者工具：

```text
F12 -> Network -> 点击页面按钮 -> 找到请求
```

看 5 个东西：

1. URL 对不对。
2. Method 是 GET 还是 POST。
3. Request Payload 或 Query String 对不对。
4. Response 里的 `code` 是多少。
5. Response 里的 `message` 是什么。

---

### 8.3 第三步：用浏览器直接测 GET

GET 接口可以直接浏览器打开。

例如：

```text
http://localhost:8080/api/admin/airport/list
http://localhost:8080/api/flight/search?departureAirportCode=PEK&arrivalAirportCode=SHA&flightDate=2026-07-01
```

如果浏览器能打开，说明后端和数据库大概率没问题，问题可能在前端调用。

---

### 8.4 第四步：看后端日志

如果返回 `50000 系统内部错误`，就看后端终端。

你要找：

- 哪个 Controller 报错。
- 哪个 Service 报错。
- 是否是空指针。
- 是否是数据库约束冲突。
- 是否是字段格式错。

不要只看前端的“系统内部错误”。

---

### 8.5 第五步：查数据库数据

如果接口没报错但数据不对，就查数据库。

常用检查：

```powershell
mysql -u root -p051130 -e "USE airticket; SELECT FlightId, FlightNumber, FlightDate, FlightStatus, DepartureAirportCode, ArrivalAirportCode FROM Flight LIMIT 20;"
```

```powershell
mysql -u root -p051130 -e "USE airticket; SELECT SegmentId, FlightId, OriginAirportCode, DestinationAirportCode, EconomyRemainingSeats, FirstClassRemainingSeats FROM FlightSegment LIMIT 20;"
```

```powershell
mysql -u root -p051130 -e 'USE airticket; SELECT UserId, LoginAccount, Points, MemberLevel FROM `User`;'
```

## 10. 你应该掌握的最小技术解释

你是后端小白也没关系，但下面这些概念你要能讲清楚。

### 10.1 什么是接口

接口就是前端和后端约定好的通信入口。

例如：

```text
POST /api/ticket/pay
```

意思是：

- 前端向这个地址发请求。
- 请求体里传 `ticketId`。
- 后端收到后执行支付逻辑。
- 后端返回统一 JSON。

---

### 10.2 什么是 DTO

DTO 就是前端和后端传数据用的对象。

例如 `CreateTicketRequest`：

```text
userId
flightId
segmentId
cabinClass
passengerName
passengerIdNumber
mealId
```

前端创建订单时字段名必须和 DTO 一致。

---

### 10.3 什么是 Entity

Entity 是 Java 代码里和数据库表对应的类。

例如：

```text
TicketSale Entity -> TicketSale 表
FlightSegment Entity -> FlightSegment 表
```

Entity 字段映射到数据库字段，所以数据库字段不能乱改。

---

### 10.4 什么是事务

事务就是一组操作要么都成功，要么都失败。

例如下单：

```text
扣库存 + 创建订单
```

这两个必须一起成功。如果订单创建失败，库存也不能被扣掉。

---

### 10.5 什么是外键

外键用来保证数据之间的关系。

例如：

```text
TicketSale.UserId -> User.UserId
```

意思是订单必须属于一个真实存在的用户。

所以不能随便删除用户，否则订单就找不到用户了。

---

### 10.6 什么是状态机

状态机就是规定状态只能按固定路线变化。

订单状态：

```text
PENDING_PAYMENT -> PAID
PENDING_PAYMENT -> EXPIRED
PAID -> REFUND_SUCCESS
PAID -> CHANGE_SUCCESS
```

不能随便加状态，也不能乱跳。
