# 航空票务系统全栈技术需求文档（交付 Codex 版）

## 1. 项目定位、目标与约束

本项目是本科数据库课程设计期末系统，目标不是商业级航空平台，而是一个能够在课堂演示中稳定拿分的航空票务系统。系统必须同时满足三个核心目标：第一，数据库结构必须规范，遵循第三范式，实体、主键、外键、唯一性约束与业务状态必须清晰；第二，核心业务必须闭环完整，至少覆盖用户注册登录、航班查询、购票、模拟支付、退票、改签、餐食预订、订单查询、管理员航班维护等流程；第三，演示必须可控稳定，所有关键流程应当有预置数据支撑，确保现场不会出现“查不到航班”“库存超卖”“支付失败”“无权限”等破坏演示的问题。

系统的部署环境固定为本地全栈部署，不使用 Docker，不依赖云服务。后端固定为 Java + Spring Boot 2.7.x，数据库固定为 MySQL 9.6，前后端通过 HTTP 进行 GET/POST 请求通信，数据载体统一为 JSON。前端技术栈不在本需求中强行锁死，后续可以再与 Codex 讨论，但前端页面结构、菜单、交互流程、接口字段规范必须在本文中一次性定义清楚。

用户上传的 ER 图是最终约束，本文所有设计必须严格贴合该 ER 图，不新增实体，不删减实体，不更改关系方向，不擅自引入新的业务表。中期汇报已经确认了基础实体层、业务层、用户层以及餐食层的分层思路，并强调了 3NF、区间售票、行级锁防超售、退改签闭环等核心点，本需求文档将以这些既有约束为基础继续细化。fileciteturn0file0L13-L15 fileciteturn0file0L23-L37

## 2. 项目需求总览

系统分为管理员端与乘客端两大角色。

管理员端负责城市、机场、飞机、航班、航段、餐食等基础数据维护，其中最重要的演示模块是航班信息的完整增删改查。由于历史票务数据必须可追溯，Flight 一旦存在票务记录，不允许物理删除，只允许将 FlightStatus 改为“停用”。乘客端负责注册、登录、航班查询、机票预订、餐食选择、模拟支付、订单管理、退票、改签、积分与会员等级展示。

系统的核心业务闭环应当是：用户注册或登录后，查询航班与航段，选择舱位与餐食，下单后锁定座位余量并生成 15 分钟支付窗，用户点击支付后直接判定为成功，成功后订单变为已支付并出票，积分自动累计；若订单过期，则库存自动回补；若用户退票，则库存回补并可同步回滚积分；若用户改签，则生成新订单并建立链式追踪，原票与新票之间通过 OriginalTicketId 关联。

系统必须支持三个预置账号：一个管理员账号、两个乘客账号。其中一个乘客账号初始积分设置为 900 分，保证其在购买一张票后积分达到 1000 分并自动升级为 VIP，再购买第二张票时自动体现九折优惠，从而形成现场最直观的会员升级演示链路。

## 3. 角色、权限与页面级菜单

系统角色仅有两类：乘客与管理员。乘客登录后只看到与本人有关的功能，包括航班查询、下单、支付、订单列表、订单详情、退票、改签、餐食选择、积分与会员等级信息。管理员登录后看到基础数据维护菜单，包括城市管理、机场管理、飞机管理、航班管理、航段管理、餐食管理，以及所有用户的积分概览和全部订单查询，但管理员没有修改用户积分的权限。

前端页面菜单必须随角色动态切换。管理员端页面上不应出现“我的订单”“积分商城”这类与乘客相关的业务入口；乘客端页面上不应出现“城市管理”“机场管理”“航班删改”等管理员入口。菜单权限的判断应由后端返回的 UserType 以及登录令牌中的角色信息共同控制，前端只负责展示，后端负责强校验。

## 4. 关键业务规则

### 4.1 航段粒度与库存模型

FlightSegment 是本系统最重要的业务粒度，定义为“可售区间”。如果某个航班的实际航线为 A → B → C，则系统中应允许出现 A→B、B→C、A→C 三种可售区间；其中 A→C 不是附加表，也不是拼接出来的临时计算结果，而是直接作为一条可售区间记录存入 FlightSegment。库存也以 FlightSegment 为最小粒度独立管理，每条可售区间拥有自己独立的 FirstClassRemainingSeats 与 EconomyRemainingSeats。

本项目明确采用“每一个可售区间独立库存”的简化方案，不做现实航空中那种跨区间联动占座，这样更符合课程设计的开发复杂度，也更容易保证演示稳定。即使同一 Flight 下存在多个可售区间，每个区间的余票也彼此独立，购票、退票、过期释放均只影响对应的 SegmentId。

### 4.2 购票与支付

一张票只对应一个航段，即 TicketSale 中的 SegmentId 只能关联一个 FlightSegment。乘客在一个订单中只能购买 A→B、B→C、A→C 中的某一个区间，不能在一张票里同时购买多个区间。如果需要多个区间，必须拆成多张票。

购票采用“先锁库存、后支付”的简化两阶段流程。用户提交下单请求后，系统立即对目标 FlightSegment 执行 `SELECT ... FOR UPDATE`，检查对应舱位是否还有余票；如有余票，立即扣减对应舱位库存并创建 TicketSale，TicketStatus 置为“待支付”，ExpiredAt = BookedAt + 15 分钟。这个时刻开始，该座位已被锁定，其他用户即使看到余票减少，也无法再购买。用户在 15 分钟内点击支付按钮后，系统直接判定支付成功，将 TicketStatus 改为“已支付”，记录 PaidAt 与 IssuedAt。支付失败不需要对接真实支付网关，支付按钮仅作为演示操作。

支付超时后的释放方式采用定时任务方案。后台定时扫描所有 TicketStatus = “待支付” 且 ExpiredAt < NOW() 的记录，将其状态改为“已过期”，同时把对应 Segment 的余票加回去。这个过程必须放在事务中执行，防止库存回补和订单状态更新出现不一致。

### 4.3 退票规则

退票规则简化为“任何时间都可以退票”，不再保留起飞前 24 小时限制，以保证演示过程简单、可控、少出错。用户对已支付订单发起退票后，系统将 TicketStatus 更新为“退款成功”，记录 RefundedAt，并将该订单对应航段的余票加回去。若系统启用了积分累计，则退票时同步扣回该票此前获得的积分，并重新计算会员等级。

### 4.4 改签规则

改签支持跨航班改签，即用户可以从一个 Flight/Segment 改到另一个 Flight/Segment。改签采用“退旧票 + 买新票”的链式追踪方式。OriginalTicketId 指向改签之前的上一张票，而不是最初那张票；如果同一行程连续改签两次，则会形成链式追踪，而不是统一回指第一张原票。

若新票价格高于旧票价格，则存在差价，差价必须在 15 分钟支付窗内补足；若新票价格不高于旧票价格，则可以视为无需补差价，直接进入改签成功流程。改签成功后，原票 TicketStatus 置为“改签成功”，记录 ChangedAt；新票按照正常购票流程创建并支付。改签同样需要先锁定新航段库存，若差价支付超时，新锁定的库存必须回补，原票保持原状态不变。

### 4.5 会员积分与 VIP 规则

会员体系设计为简单、可演示、可解释。用户每成功支付 1 张机票，自动获得 100 积分；当 Points 达到 1000 分时，MemberLevel 自动升级为 VIP；当积分低于 1000 分时，MemberLevel 为普通。数据库只存原价，TicketSale 中的 PriceAmount 存原始票面价格，PaymentAmount 存实际支付金额。VIP 用户购票时，在原价基础上乘以 0.9 作为实际支付金额，数据库仍然保存原价和折后实际支付额，便于演示“原价不变、支付额变化”的业务逻辑。

为了演示会员升级，预置乘客 A 的初始积分为 900 分。该用户在演示时先购买一张票，积分自动到达 1000 分并升级为 VIP；随后再购买一张票，支付金额自动按 0.9 折计算，界面必须能直观看到优惠生效。

### 4.6 身份证号与密码安全

密码必须使用 BCrypt 哈希存储，身份证号必须使用不可逆加密处理。为降低实现复杂度并确保可落地，建议密码字段 PasswordHash 在后端使用 BCrypt 直接生成；身份证号相关字段 IdNumberDigest 与 PassengerIdNumberDigest 在后端使用固定盐 + SHA-256 摘要生成。前端传输阶段可以继续使用同名字段提交原始值，后端收到后立即做哈希，再写入数据库。这样既满足“JSON 字段名严格一致”的要求，也满足不可逆存储的要求。

## 5. 数据库设计总原则

数据库设计必须严格遵循第三范式。每张表只表达一个主题，非主键字段必须完全依赖主键，不能有部分依赖和传递依赖。Flight 只存航班级信息，不再存计划/实际起降时刻与延误信息；这些信息由 FlightSegment 承担。User 只存用户身份、登录、会员与联系信息；票务事实由 TicketSale 表承载。MealOption 只存餐食定义，MealReservation 只存餐食与票的关联。这样设计既符合 3NF，也符合你们中期汇报中“无数据冗余、支持区间售票、便于历史追溯”的主张。fileciteturn0file0L24-L25

### 5.1 实体与关系清单

系统共包含 9 个实体：City、Airport、Aircraft、Flight、FlightSegment、User、TicketSale、MealOption、MealReservation。其关系如下：City 与 Airport 为一对多；Aircraft 与 Flight 为一对多；Flight 与 FlightSegment 为一对多；Airport 与 Flight、FlightSegment 分别通过起飞/到达机场码建立外键关联；User 与 TicketSale 为一对多；Flight 与 TicketSale 为一对多；FlightSegment 与 TicketSale 为一对多；TicketSale 与 TicketSale 之间通过 OriginalTicketId 建立自引用链式关联；TicketSale 与 MealReservation 为一对多；MealOption 与 MealReservation 为一对多。你们中期汇报已经明确了“航班→航段”“航段→售票记录”“售票记录→餐食选项”的主干关系，这里在不改变结构的前提下只做落地化细化。fileciteturn0file0L15-L15

### 5.4 索引设计

City 建议对 CityCode 建唯一索引。Airport 建议对 CityId 建普通索引，便于按城市查机场。Flight 建议对 FlightNumber + FlightDate 建联合唯一索引，对 AircraftRegNo、DepartureAirportCode、ArrivalAirportCode、FlightStatus 建普通索引，对 FlightDate 建普通索引以支持日期范围查询。FlightSegment 建议对 FlightId、OriginAirportCode、DestinationAirportCode、PlannedDepartureTime 建索引，并对 FlightId + OriginStopNo + DestinationStopNo 建联合唯一索引，以保证同一航班下区间记录不重复。User 建议对 LoginAccount、IdNumberDigest 建唯一索引。TicketSale 建议对 OrderNo 唯一，对 UserId、FlightId、SegmentId、TicketStatus、OriginalTicketId、ExpiredAt 建索引。MealOption 建议对 MealName 唯一。MealReservation 建议对 TicketId + MealId 建联合唯一索引。

### 5.5 完整性约束

实体完整性要求所有主键非空且唯一；参照完整性要求所有外键均存在有效父记录，删除父表时若存在子记录则禁止物理删除；业务完整性要求余票不得为负，起飞到达时间顺序不得颠倒，改签链必须可追踪，状态字段只能在白名单中切换。你们中期汇报已经强调了“主键非空唯一、身份证号加密唯一、所有外键级联更新/限制删除、售票时锁定行级记录、已使用机票不可退改签”等完整性要求，这里将其落到文档化实现规则中。fileciteturn0file0L36-L37

## 6. 事务、并发与锁控制

购票、退票、改签、过期释放库存都必须放在事务中执行，至少使用 `@Transactional` 包裹 Service 方法。系统采用 MySQL 行级锁解决超售问题，核心语句是对目标 FlightSegment 执行 `SELECT ... FOR UPDATE`，锁定后再检查余票，再扣减余票，再插入票务记录。你们中期汇报已经明确“原子性事务保证并通过行级锁解决并发超售问题”“检查余票、扣减余票、超时回滚、行级锁定、生成订单”等关键过程，这里将其固化为必须实现的开发要求。fileciteturn0file0L27-L30

建议采用以下事务边界：

1. 下单事务：锁定 Segment 行，判断库存，扣减库存，创建 TicketSale，创建 MealReservation，写入 BookedAt 与 ExpiredAt。
2. 支付事务：检查 TicketStatus 是否仍为待支付且未过期，更新为已支付，写入 PaidAt 与 IssuedAt，积分增加 100，更新会员等级。
3. 过期扫描事务：扫描超时待支付订单，更新为已过期，恢复库存。
4. 退票事务：校验 TicketStatus 为已支付或允许退票的状态，恢复库存，更新为退款成功，回滚积分并重算会员等级。
5. 改签事务：创建新 TicketSale 并锁定新库存，若存在差价则进入 15 分钟支付窗，支付完成后更新原票状态为改签成功，恢复原票库存，保留新票库存。

数据库隔离级别建议使用 `READ COMMITTED` 或在默认 `REPEATABLE READ` 下结合 `FOR UPDATE` 做显式行锁控制。考虑到课程设计重在可解释与稳定，推荐以 `FOR UPDATE` 为主，避免依赖复杂的间隙锁逻辑。定时任务建议每 1 分钟扫描一次待支付超时订单，防止演示时出现过期单长时间不回收的问题。

## 7. 后端系统设计

### 7.1 总体架构

后端采用 Spring Boot 2.7.x 三层架构：Controller → Service → Repository（或 Mapper）。Controller 负责接收 HTTP 请求、参数校验、响应封装；Service 负责业务编排、事务控制、状态流转、积分与库存计算；Repository 负责访问 MySQL，执行普通查询与 `FOR UPDATE` 锁定查询。所有查询类接口通过 GET 暴露，所有会改变数据库状态的接口通过 POST 暴露，不使用 PUT/DELETE，以减少课堂联调和前端实现复杂度。

推荐的后端分层包结构如下：

- `controller`：AuthController、UserController、CityController、AirportController、AircraftController、FlightController、FlightSegmentController、TicketController、MealController、AdminController
- `service`：各模块业务接口
- `service.impl`：业务实现
- `repository` 或 `mapper`：数据库访问层
- `entity`：与数据库表一一对应的实体类
- `dto`：请求数据对象
- `vo`：响应视图对象
- `enums`：UserType、MemberLevel、FlightStatus、TicketStatus、CabinClass 等
- `scheduler`：支付超时扫描任务
- `config`：CORS、Swagger/OpenAPI、MyBatis 等配置
- `exception`：统一异常与错误码

### 7.2 登录认证与权限控制

登录成功后，后端返回 token 与用户基础信息。建议采用 JWT 作为本地前后端分离的轻量认证方式，前端将 token 保存到本地存储，并在后续请求中通过 `Authorization: Bearer xxx` 携带。后端根据 token 解析 UserId、UserType、MemberLevel，用于菜单权限与业务校验。管理员与乘客仅在登录后权限不同，不需要另建权限表，避免超出课程设计复杂度。

注册接口只允许创建乘客账号，UserType 固定为“乘客”，管理员账号只能由初始化脚本预置。注册时需要校验 LoginAccount 唯一、手机号和邮箱格式、密码强度、身份证号唯一。由于数据库存储的是 `PasswordHash` 与 `IdNumberDigest`，前端提交的同名字段值可以是明文，后端在服务层进行哈希后落库。

### 7.3 核心服务逻辑

航班查询服务需要以城市维度查询。前端传入 DepartureCityId、ArrivalCityId、StartDate、EndDate、CabinClass，后端先通过 CityId 找到对应机场集合，再联查 Flight 与 FlightSegment，筛选出满足条件的可售区间，并按起飞时间排序返回。查询结果应展示 FlightNumber、FlightDate、OriginAirportCode、DestinationAirportCode、PlannedDepartureTime、PlannedArrivalTime、CabinClass 对应的库存、价格、FlightStatus。

购票服务需要先验证用户身份、航班与区间是否存在、余票是否充足、航班状态是否可售。锁库存后生成订单号 OrderNo，写入 TicketSale。若乘客已选择餐食，则同时创建 MealReservation。支付服务只做模拟成功，不连接任何真实支付 SDK；支付成功后立即写入 PaidAt 与 IssuedAt，并发放积分。退票服务需要检查订单是否已支付、航班是否允许退票、订单是否未过期，再执行库存回补和状态更新。改签服务则需要先校验旧票状态，再创建新票与新库存锁定记录，若需要补差价则生成差额支付流程，支付完成后把原票状态改为改签成功。

### 7.4 统一响应格式

所有接口统一返回如下结构：

```json
{
  "success": true,
  "code": 200,
  "message": "success",
  "data": { }
}
```

当发生错误时，`success` 为 false，`code` 使用自定义业务码，`message` 给出可读错误信息。`data` 内部的业务字段名必须严格对应数据库字段名，例如 Flight、TicketSale、User 的返回对象都应使用与表字段一致的命名，避免大小写、拼写和字段映射错误。

### 7.5 关键接口设计

以下接口仅使用 GET 与 POST，字段名与数据库字段保持一致。

#### 7.5.1 认证模块

`POST /api/auth/register`  
请求体字段：LoginAccount、UserName、PasswordHash、IdNumberDigest、PhoneNumber、Email。  
业务说明：创建乘客账号，UserType 自动写为“乘客”，Points 默认 0，MemberLevel 默认“普通”。后端对 PasswordHash 和 IdNumberDigest 做哈希处理后入库。  
响应数据：UserId、LoginAccount、UserName、UserType、Points、MemberLevel、CreatedAt。

`POST /api/auth/login`  
请求体字段：LoginAccount、PasswordHash。  
业务说明：PasswordHash 字段在请求阶段承载明文密码，后端用 BCrypt 校验。  
响应数据：token、UserId、LoginAccount、UserName、UserType、Points、MemberLevel。

`GET /api/auth/me`  
请求参数：无。  
业务说明：根据 token 返回当前登录用户信息。  
响应数据：UserId、LoginAccount、UserName、UserType、Points、MemberLevel、CreatedAt、UpdatedAt。

#### 7.5.2 城市与机场模块

`GET /api/city/list` 返回全部可用城市。  
`POST /api/city/save` 新增或修改城市，字段：CityId、CityName、CityCode、Country。  
`POST /api/city/delete` 仅允许未被机场引用的城市删除。  

`GET /api/airport/list` 支持按 CityId 查询机场。  
`POST /api/airport/save` 新增或修改机场，字段：AirportCode、AirportName、CityId、IsInternational。  
`POST /api/airport/delete` 仅允许未被航班或航段引用的机场删除。

#### 7.5.3 飞机模块

`GET /api/aircraft/list` 查询飞机。  
`POST /api/aircraft/save` 新增或修改飞机，字段：AircraftRegNo、AircraftType、Manufacturer、TotalFirstClassSeats、TotalEconomySeats、Status、Remark。  
`POST /api/aircraft/delete` 仅允许未被航班引用的飞机删除。  

`Status` 建议支持“在役”“停用”“检修中”。

#### 7.5.4 航班与航段模块

`GET /api/flight/list` 按 FlightNumber、FlightDate、DepartureAirportCode、ArrivalAirportCode、FlightStatus 查询。  
`POST /api/flight/save` 新增或修改航班，字段：FlightId、FlightNumber、FlightDate、AircraftRegNo、FlightStatus、DepartureAirportCode、ArrivalAirportCode、Remark。  
业务规则：若 Flight 已存在 TicketSale 记录，则禁止修改 FlightNumber、FlightDate、AircraftRegNo、DepartureAirportCode、ArrivalAirportCode，只允许修改 FlightStatus 和 Remark。  
`POST /api/flight/disable` 将 FlightStatus 设为“停用”，不做物理删除。  
`POST /api/flight/delete` 仅允许无票务记录时物理删除；一旦存在票务记录，一律禁止。

`GET /api/flight-segment/list` 按 FlightId 或起降机场查询。  
`POST /api/flight-segment/save` 新增或修改航段，字段：SegmentId、FlightId、OriginStopNo、DestinationStopNo、OriginAirportCode、DestinationAirportCode、PlannedDepartureTime、PlannedArrivalTime、ActualDepartureTime、ActualArrivalTime、DelayMinutes、DelayReason、FirstClassRemainingSeats、EconomyRemainingSeats、FirstClassPrice、EconomyPrice、Remark。  
业务规则：同一 FlightId 下 OriginStopNo 与 DestinationStopNo 的组合必须唯一；OriginStopNo 必须小于 DestinationStopNo；票价与库存不得为负数。

#### 7.5.5 航班查询与购票模块

`GET /api/flight/search`  
请求参数：DepartureCityId、ArrivalCityId、StartDate、EndDate、CabinClass。  
业务说明：按城市而非单机场搜索，后端自动展开 CityId 对应的机场集合，查找可售航段。  
响应数据：FlightId、FlightNumber、FlightDate、AircraftRegNo、FlightStatus、DepartureAirportCode、ArrivalAirportCode、SegmentId、OriginStopNo、DestinationStopNo、OriginAirportCode、DestinationAirportCode、PlannedDepartureTime、PlannedArrivalTime、FirstClassRemainingSeats、EconomyRemainingSeats、FirstClassPrice、EconomyPrice。

`POST /api/ticket/book`  
请求体字段：UserId、FlightId、SegmentId、CabinClass、PassengerName、PassengerIdNumberDigest、MealId、Quantity、ChangeReason。  
业务说明：订单创建接口。后端加锁扣减库存，创建 TicketSale，状态置为待支付，ExpiredAt 写入 15 分钟后时间。若选择餐食，则创建 MealReservation。  
响应数据：TicketId、OrderNo、TicketStatus、PriceAmount、PaymentAmount、BookedAt、ExpiredAt。

`POST /api/ticket/pay`  
请求体字段：TicketId。  
业务说明：模拟支付成功，检查订单未过期后直接置为已支付，写入 PaidAt、IssuedAt，同时根据是否 VIP 计算 PaymentAmount，增加 100 积分并更新 MemberLevel。  
响应数据：TicketId、OrderNo、TicketStatus、PaymentAmount、PaidAt、IssuedAt、UserId、Points、MemberLevel。

`POST /api/ticket/expire-process`  
请求体字段：可空，定时任务内部调用。  
业务说明：扫描超时待支付订单，更新为已过期并回补库存。

`POST /api/ticket/refund`  
请求体字段：TicketId、Remark。  
业务说明：任何时间均可退票。成功后状态置为退款成功，写入 RefundedAt，回补库存，并扣回该票已获得的积分。  
响应数据：TicketId、TicketStatus、RefundedAt、UserId、Points、MemberLevel。

`POST /api/ticket/change`  
请求体字段：TicketId、FlightId、SegmentId、CabinClass、PassengerName、PassengerIdNumberDigest、MealId、Quantity、ChangeReason。  
业务说明：生成新 TicketSale 作为改签目标，OriginalTicketId 指向原票。若新票价格高于原票，则返回差额支付窗；支付成功后原票状态改为改签成功，新票状态改为已支付。若差额为 0，则可直接完成。  
响应数据：OldTicketId、NewTicketId、OrderNo、PriceAmount、PaymentAmount、ExpiredAt、TicketStatus。

#### 7.5.6 订单、餐食与用户模块

`GET /api/ticket/list-by-user` 按 UserId 查询个人订单。  
`GET /api/ticket/detail` 按 TicketId 查询订单详情，必须返回 Flight、FlightSegment、MealReservation、MealOption、User 的关联信息。  

`GET /api/meal/list` 查询餐食选项，默认返回普通餐、清真餐、素食餐。  
`POST /api/meal/save` 新增或修改餐食，字段：MealId、MealName、MealType、IsAvailable、Description。  
`POST /api/meal/delete` 仅允许未被订单引用时删除。  

`GET /api/user/list` 管理员查看全部用户。  
`GET /api/user/detail` 查看用户详情，返回 Points 与 MemberLevel，但不允许管理员修改 Points。  
`POST /api/user/save` 一般仅用于用户注册后的资料补充，不允许修改 UserType。  
`POST /api/user/update-profile` 允许用户修改 UserName、PhoneNumber、Email。

### 7.6 字段映射特殊说明

凡是以 `Hash` 或 `Digest` 结尾的字段，前端传输阶段可以提交原始明文值，但字段名仍保持与数据库字段一致，后端在服务层统一哈希后入库。这样既满足“字段名严格一致”的要求，也保证安全性。此规则适用于 `PasswordHash`、`IdNumberDigest`、`PassengerIdNumberDigest`。前端不可直接展示哈希值，列表页与详情页应隐藏这些字段。

## 8. 前端系统设计要求

前端技术栈不在本需求中强制限定，但必须满足以下要求：页面为现代化单页应用，支持组件化开发、路由切换、表单校验、表格分页、弹窗确认、统一主题样式。视觉风格建议采用航空主题蓝色系，整体布局简洁、页面留白适度、表格与表单统一，避免花哨特效干扰演示。

### 8.1 页面结构

乘客端至少包含以下页面：登录页、注册页、首页搜索页、搜索结果页、航班详情/下单页、支付确认页、我的订单页、订单详情页、退票弹窗、改签弹窗、个人中心页。管理员端至少包含以下页面：管理员首页、用户管理页、城市管理页、机场管理页、飞机管理页、航班管理页、航段管理页、餐食管理页、订单查询页。

### 8.2 页面交互流程

登录页输入账号密码后，根据 UserType 自动跳转到不同首页。首页搜索页先选择出发城市、到达城市、日期范围和舱位，再点击查询。结果页列表展示满足条件的 FlightSegment，用户点击“预订”进入下单页，选择乘客姓名、身份证、餐食后提交。下单成功后跳转到订单详情或我的订单页，页面显示倒计时 15 分钟。支付按钮点击后直接返回成功，不接真实支付。退票、改签都必须先进入订单详情页，再弹出操作框，避免误操作。

### 8.3 前后端通信说明

前端所有查询类请求使用 GET，请求参数名与数据库字段名保持一致。所有创建、修改、删除、支付、退票、改签类请求使用 POST，请求体 JSON 字段名与数据库字段名保持一致。前端只负责展示和提交，不自行做业务判断；所有库存校验、状态校验、积分计算、会员升级、改签链维护都必须以后端返回结果为准。若后端返回 4xx/5xx，前端应以弹窗或表单提示方式清晰展示错误原因。

## 9. 管理员演示设计

管理员演示是老师最容易直接给分的部分，必须设计成“新增 → 查询验证 → 修改 → 再次查询 → 停用或删除 → 最终确认”的标准流程。最建议的演示对象是 Flight。先新增一条航班，页面显示成功后立即查询；然后修改 FlightStatus 或 Remark，再查询验证修改结果；接着将该航班停用，再次查询确认列表变化；如果该航班尚无票务记录，则可以演示删除按钮；如果已有票务记录，则演示“禁止物理删除”的约束提示。整个流程必须让老师看到数据库状态发生了可验证的变化，而不是只看表面表单。

管理员还可以演示城市、机场、飞机、餐食的基础数据维护，但主打亮点应放在航班与航段上。管理员可以查看所有用户的积分数值，但不能编辑积分，这一点应明确写入权限说明并在界面上做灰化处理。

## 10. 乘客演示设计

乘客演示推荐按以下顺序进行：先登录普通乘客账号，执行航班查询，选择某一城市对与日期范围，确认结果列表不为空；然后进入下单页，选择一个舱位和一种餐食，提交订单并展示 15 分钟倒计时；再点击支付，展示订单状态从待支付变为已支付、座位余量减少、积分增加。接着切换到预置 900 分的用户账号，先购买一张票使积分达到 1000 分并自动升级为 VIP，再购买第二张票展示 0.9 折优惠，老师可以在金额与会员等级变化中直接看到效果。若时间允许，再演示改签和退票，形成完整业务闭环。

## 11. 测试数据与种子数据方案

### 11.1 设计目标

种子数据必须保证演示当天 2026 年 7 月 1 日前后三天，即 2026-06-28 到 2026-07-04，查询一定有结果，不允许出现“查无航班”。因此核心可搜索城市应采用稳定且互通的 5 个城市，并为这 5 个城市构造完全覆盖的航班组合。为了符合“可适量添加境外机场”这一要求，可以另外加入少量国际扩展数据，但默认搜索页只展示核心城市集合，避免现场不稳定。

### 11.2 核心城市与机场

建议核心城市固定为北京、上海、广州、深圳、成都。对应机场可设置为：

- 北京：北京首都机场、北京大兴机场
- 上海：上海虹桥机场、上海浦东机场
- 广州：广州白云机场
- 深圳：深圳宝安机场
- 成都：成都天府机场、成都双流机场

如需扩展，可补充少量国际城市与机场，但默认搜索下拉应优先使用以上核心城市，以确保最稳演示。

### 11.3 航班覆盖规则

为保证任意选择一个出发地与目的地都能查到航班，核心城市之间应至少构造 20 条直达航班，覆盖 5 个城市之间所有有序城市对。具体来说，5 个城市两两不同的有序组合共 20 对，每一对至少安排 1 条 Flight 记录与 1 条对应的 FlightSegment 记录。同时再增加 2 条多航段示范航班，用于展示 A→B→C 场景下 A→B、B→C、A→C 三种可售区间均存在的能力。这样既能保证查询稳定，又能证明 FlightSegment 的设计不是单点直连，而是真正支持区间售卖。

直达航班建议按以下模式构造：

北京→上海，北京→广州，北京→深圳，北京→成都；  
上海→北京，上海→广州，上海→深圳，上海→成都；  
广州→北京，广州→上海，广州→深圳，广州→成都；  
深圳→北京，深圳→上海，深圳→广州，深圳→成都；  
成都→北京，成都→上海，成都→广州，成都→深圳。

再补充两条多航段示范航班，例如北京→上海→成都、广州→深圳→北京，用于生成 A→C 区间。所有航班日期分布在 2026-06-28 到 2026-07-04 之间，时间点可以采用 08:00、11:30、15:00、18:30、22:00 等固定时刻循环生成，便于排查和演示。

### 11.4 飞机、用户与餐食种子数据

Aircraft 建议至少 6 架，覆盖不同机型与座位容量。用户建议至少 3 个预置账号：1 个管理员、2 个乘客，其中 1 个乘客初始积分 900 分，另 1 个乘客积分 0。餐食固定为 3 项：普通餐、清真餐、素食餐。所有预置数据应在初始化脚本中一次性写入，便于老师现场重置环境。

### 11.5 种子数据导入顺序

推荐导入顺序为：City → Airport → Aircraft → User → MealOption → Flight → FlightSegment → 可选的初始 TicketSale。由于 TicketSale 是业务事实表，一般建议演示前不预置大量订单，只保留少量测试记录或完全为空，以免影响现场演示。

## 12. 部署与环境要求

系统必须本地运行，数据库端口固定 3306，后端端口固定 8080。前端与后端分离运行时，需要正确配置 CORS，允许前端页面访问后端接口。后端启动前必须先完成数据库建表与初始化数据导入。建议提供一键初始化 SQL 脚本，包括建库、建表、索引、约束、基础数据和预置账号。

系统运行检查清单应至少包含以下内容：数据库能成功连接；登录接口可用；城市与机场可查询；航班列表有数据；乘客下单可成功生成订单；支付可成功；定时任务可回收过期订单；管理员可完成航班增删改查；会员升级与九折优惠可正常演示；退票与改签可正常演示。

## 13. 交付给 Codex 的实现优先级

Codex 实现时的优先级建议如下：先建数据库表结构与初始化数据，再做实体类与 Repository/Mapper，再做 Service 层事务与锁定逻辑，再做 Controller 接口与统一响应，再做前端页面与路由，最后补充定时任务、异常处理、权限拦截和演示数据校验。任何涉及库存、状态、积分、会员等级、改签链的逻辑都必须放在后端 Service 层，不可放到前端临时拼接。

本系统最终的判断标准不是“看起来功能多”，而是“演示时能不能稳定地把数据库设计、事务控制、业务闭环、前后端联调完整展示出来”。只要按照本文实现，系统就能同时满足数据库课程设计的规范性、完整性与可演示性。
