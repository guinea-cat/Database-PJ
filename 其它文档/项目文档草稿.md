## 1 项目基本信息

本项目为航空票务数据库系统，属于数据库课程设计项目。项目以民航客运业务为背景，以航空票务业务流程为主线，围绕航空公司票务系统中的核心数据对象及其关联关系展开设计与实现。系统以数据库为核心，对旅客购票、航班组织、订单管理等业务过程进行抽象，建立完整的数据模型，并在此基础上实现典型数据库应用功能，以验证数据库设计理论在实际业务场景中的应用。

系统主要涉及用户、城市、机场、飞机、航班、航段、订单、餐食、会员积分等多个实体，并通过主键、外键及完整性约束建立实体之间的联系，构建了较为完整的航空票务业务数据体系。在业务层面，系统支持旅客注册登录、航班查询、在线购票、订单查询、退票改签、餐食预订以及会员积分管理等功能；同时支持管理员对城市、机场、飞机、航班、航段、餐食等基础数据进行维护，并能够对用户和订单信息进行统一管理，从而保证系统数据的一致性、完整性与可维护性。

系统采用前后端分离架构，但整体设计以数据库为核心。数据库负责存储和管理各类业务数据；后端负责业务逻辑处理、事务控制以及数据库访问；前端负责用户界面展示和交互操作。在系统运行过程中，前端通过 HTTP 请求调用后端接口，后端根据业务需求对 MySQL 数据库进行增删改查操作，并将查询结果以 JSON 格式返回前端，实现数据层、业务层与表示层之间的有效协同。

项目工程主要由四部分组成：database 模块用于存放数据库建表脚本、约束定义及初始化数据；backend 模块基于 Spring Boot 实现，负责业务逻辑处理和数据库访问；frontend 模块基于 React 实现，用于构建旅客端和管理员端界面；docs 模块用于存放项目设计文档、接口文档及相关说明材料。

## 2 系统需求分析

## 2.1 系统目标

本系统以航空票务业务为应用背景，旨在设计并实现一个面向航空客运场景的数据库应用系统。系统以 MySQL 数据库为核心，对航空票务业务中的用户、航班、航段、订单、飞机、机场等数据进行统一管理，实现航空票务业务全过程的信息化处理。

系统的主要目标包括：

1. 建立完整的航空票务业务数据模型，实现核心业务数据的集中存储与统一管理；

2. 支持旅客完成注册登录、航班查询、在线购票、订单支付、退票改签等基本业务操作；

3. 支持管理员对城市、机场、飞机、航班、航段、餐食等基础数据进行维护；

4. 利用主键、外键、唯一约束以及业务规则约束保证数据的一致性与完整性；

5. 通过事务机制实现库存、订单状态和会员积分等关键数据的同步更新；

6. 对用户密码、身份证号等敏感信息进行安全保护，提高系统安全性。

## 2.6 主要业务流程

系统核心业务包括购票、退票和改签三个过程，其总体业务流程如下：

1. 旅客登录系统后，根据出发地、目的地和日期查询可售航段；

2. 用户选择目标航段、舱位和餐食信息后提交订单，系统生成待支付订单并扣减对应库存；

3. 用户在规定时间内完成支付，支付成功后系统更新订单状态并累计会员积分；

4. 已支付订单允许进行退票或改签，系统在业务处理完成后同步更新库存、订单状态和积分信息；

5. 管理员负责维护基础业务数据，并定期处理超时未支付订单，保证系统数据的正确性和完整性。

## 3.3 概念模型设计特点

与传统票务系统直接将整条航班作为售票单位不同，本系统引入了 FlightSegment（航段）实体，并将其设计为系统实际售卖的最小业务单位。例如，对于执行路线为 A-B-C 的航班，系统不仅能够销售 A-B 和 B-C 两个区间，还能够销售 A-C 区间，从而支持经停航班的区间化售票。

采用航段级销售模型具有以下优势：首先，可以对不同区间分别设置票价和库存，提高系统对实际航空业务场景的描述能力；其次，库存扣减、退票回补以及超时订单处理均能够精确到具体航段，避免不同区间之间的库存冲突；最后，改签、订单统计等业务操作也能够直接关联到具体航段，从而增强数据库模型的可扩展性和业务表达能力。

因此，航段实体是本系统概念模型设计中的核心实体，也是整个数据库设计区别于传统简化模型的重要特征。

## 7 系统实现说明

本系统采用前后端分离架构实现业务功能。前端通过 HTTP 接口与后端进行通信，后端基于 Spring Boot 框架实现业务逻辑，并通过 Spring Data JPA 访问 MySQL 数据库。系统重点实现了用户管理、航班查询、订单处理、库存控制和会员管理等核心业务。

## 7.1 统一接口设计

为了提高系统接口的一致性和可维护性，后端所有接口均采用统一的数据返回格式，并使用 ApiResponse 对返回结果进行封装。系统接口统一以前缀

```txt
http://localhost:8080/api 
```

进行访问，其中查询类操作主要采用 GET 请求，状态修改类操作主要采用 POST 请求。后端统一返回的数据格式如下：

```json
{
    "code": 0,
    "message": "success",
    "data": {},
    "timestamp": "2026-07-01T12:00:00"
} 
```

其中，code 表示业务执行状态，message 表示提示信息，data 用于封装业务数据，timestamp 表示接口响应时间。前端通过 api/client.ts 对接口请求、异常信息和错误提示进行统一处理。

## 7.2 用户管理实现

用户注册与登录功能由认证模块负责实现。注册过程中，系统首先对账号、密码、姓名、手机号、邮箱和身份证号格式进行合法性校验；随后检查用户账号和身份证摘要是否已经存在，以保证用户身份的唯一性。出于安全性考虑，系统不直接保存用户密码和身份证号明文，而是分别采用 PasswordHash 和 IdNumberDigest 字段保存其摘要值。新注册用户默认会员等级为 NORMAL，初始积分为 0。当用户选择注销账号时，系统不会物理删除用户记录，而是采用匿名化处理方式隐藏用户身份信息，同时保留历史订单记录，以保证票务数据的完整性和可追溯性。

## 7.3 航班查询实现

系统提供基于出发地、目的地和日期的组合查询功能。航班查询模块仅返回当前处于可销售状态的航班数据，其中状态为 NORMAL 和 DELAYED 的航班允许继续售票。与传统票务系统不同，本系统采用航段级销售模式，系统实际查询和展示的对象均为FlightSegment 实体。用户最终购买的也是具体航段，而非整条航班。

## 7.4 订单处理实现

订单处理是系统的核心业务模块，主要包括下单、支付、退票和改签四个过程。

在用户下单时，系统首先锁定目标航段数据并检查对应舱位库存；若库存充足，则扣减库存并创建状态为 PENDING_PAYMENT 的待支付订单，同时记录订单创建时间和订单失效时间。系统默认支付时限为 15 分钟。

当用户选择餐食时，系统同时向 MealReservation 表中写入对应的餐食预订记录。

支付时，系统首先锁定目标订单，并校验订单当前状态是否仍为待支付状态。若订单已经超时，则系统自动将订单状态更新为 EXPIRED 并恢复库存；若支付成功，则将订单状态更新为 PAID，同时为用户增加积分。当积分累计达到 1000 分时，系统自动将用户会员等级升级为 VIP。

退票业务仅允许对已支付订单执行。退票成功后，系统自动恢复对应航段库存，并同步扣回积分，随后重新计算用户会员等级，最终将订单状态更新为 REFUND_SUCCESS。

改签业务同样仅允许针对已支付订单进行处理。系统通过创建新的订单记录完成改签，并利用 OriginalTicketId 字段建立新旧订单之间的关联关系。改签过程中，系统重新扣减新航段库存，并根据差价计算最终支付金额。改签完成后，原订单状态变更为CHANGE_SUCCESS。

## 7.5 订单状态机设计

为规范订单生命周期，系统采用状态机机制管理订单状态。支持的订单状态包括：

• PENDING_PAYMENT：待支付；

• PAID：已支付；

• EXPIRED：订单已过期；

• REFUND_SUCCESS：退票成功；

• CHANGE_SUCCESS：改签成功。

订单状态之间的流转关系如图 ?? 所示：

```txt
PENDING_PAYMENT -> PAID
PENDING_PAYMENT -> EXPIRED
PAID -> REFUND_SUCCESS
PAID -> CHANGE_SUCCESS 
```

系统严格限制订单状态转换路径，避免出现非法状态迁移。

## 7.6 库存控制与事务管理

系统未单独设计库存表，而是直接在 FlightSegment 实体中维护不同舱位的剩余座位数量。为了保证库存数据的一致性，所有涉及库存变化的业务操作均采用事务机制进行管理。在下单、支付、退票、改签以及过期订单处理过程中，系统首先对相关航段或订单记录加锁，然后在同一事务中完成库存更新、订单状态更新以及积分变更等操作。通过事务机制和数据库行级锁，我们有效避免了超卖问题以及“库存已修改但订单未更新”等中间状态的出现。

## 7.7 定时任务实现

系统实现了超时订单自动处理机制。后台定时任务 TicketExpireScheduler 每分钟执行一次，自动扫描所有超过支付期限但尚未支付的订单。对于超时订单，系统自动将其状态更新为 EXPIRED，并恢复对应航段库存，从而保证库存数据的准确性，提高系统自动化管理能力。

## 7.8 数据安全与隐私保护

我们高度重视用户隐私数据的安全性。用户密码采用哈希方式存储，身份证号仅保存摘要信息，数据库和前端页面均不保存或展示敏感信息明文。此外，管理员页面仅允许查看必要的用户信息，不展示 PasswordHash、IdNumberDigest 等敏感字段，从而进一步降低用户隐私泄露风险。

## 11 总结与展望

本项目以航空票务业务场景为背景，设计并实现了一个基于 MySQL、Spring Boot和 React 的航空票务数据库系统。系统围绕旅客购票和管理员维护两条业务主线，完成了从需求分析、概念模型设计、关系模式设计到系统实现与测试的完整开发过程。

在数据库设计方面，我们以 ER 模型为基础，对城市、机场、飞机、航班、航段、用户、订单和餐食等核心实体进行了建模，并通过主键、外键、唯一约束以及检查约束保证数据的完整性与一致性。针对航空票务业务特点，我们创新采用航段级销售模式，实现了经停航班不同区间的独立售卖和库存管理，提高了数据模型的业务表达能力。

在系统实现方面，我们采用前后端分离架构，利用 Spring Boot 实现业务逻辑处理，利用 React 实现用户交互界面，并通过统一接口规范完成前后端通信。我们重点实现了航班查询、在线购票、支付、退票、改签、会员积分管理以及后台资源维护等核心功能。同时，我们通过事务机制、行级锁和订单状态机有效保证了订单、库存和积分数据的一致性，较好地体现了数据库课程设计中关于事务管理和数据完整性的核心思想。

总体来看，本系统已经能够完整支持航空票务的基本业务流程，达到了课程设计的预期目标。未来可以考虑在以下几个方面进一步扩展和优化：

• 引入真实支付接口，实现更加完整的在线支付流程；

• 增加航班实时动态、延误预警等功能，提高系统实用性；

• 引入权限控制框架，实现更细粒度的角色权限管理；

• 增加统计分析与数据可视化功能，为运营决策提供支持；

• 针对高并发购票场景进一步优化库存控制策略，提高系统性能。

通过本次课程设计，我们不仅加深了对数据库建模、关系模式设计、事务管理和完整性约束等理论知识的理解，也提升了综合运用数据库技术解决实际问题的能力。
